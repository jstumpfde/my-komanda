# ТЗ-4: UX багфиксы анкеты и демо-страницы

**Цель:** Починить 4 бага в анкете кандидата и на странице прохождения демо.
**Оценка:** ~15-20 минут.

---

## БАГИ

### Баг 1: Поля анкеты белые вместо серых

**Симптом:** На странице анкеты кандидата (после прохождения демо) поля ввода
имеют слишком яркий placeholder — текст «Введите имя» выглядит почти как
введённое значение, путает кандидата. Должно быть приглушённо-серым.

**Где искать:**
- components/anketa/* или components/candidate-form/*
- Страница: /demo/[token] финальная анкета
- Возможно в demo-client.tsx или в отдельном компоненте AnketaForm

**Что сделать:**
1. Найди компонент анкеты после демо
2. У <input> или <Input> найди класс placeholder:text-* или style placeholder
3. Заменить яркий цвет на text-muted-foreground / text-gray-500 / text-zinc-500
4. Если placeholder вообще не стилизован — добавить placeholder:text-muted-foreground

**Тест:** Поля анкеты должны выглядеть пустыми (серый placeholder), а не
заполненными.

---

### Баг 2: Дата рождения — текстбокс с маской вместо календаря

**Симптом:** Сейчас поле «Дата рождения» в анкете — простой текстбокс с
маской дд.мм.гггг. Кандидаты путаются. Нужен календарь (date picker).

**Где искать:**
- Тот же компонент анкеты что и Баг 1
- Поле name="birthDate" / "birth_date" / "дата_рождения"

**Что сделать:**
1. Заменить <Input type="text" /> на компонент date picker
2. Использовать существующий компонент календаря из shadcn/ui если он уже есть
   в проекте (components/ui/calendar.tsx, components/ui/date-picker.tsx)
3. Если такого нет — использовать <Input type="date" /> как минимальный вариант
   (нативный браузерный date picker, без новых библиотек)
4. Формат значения в БД оставить как есть (ISO YYYY-MM-DD скорее всего)

**ВАЖНО:** Не ставь новые библиотеки (react-datepicker и т.п.). Если в проекте
уже есть Calendar/DatePicker из shadcn — используй его. Если нет —
<Input type="date" />.

**Тест:** Кликаешь по полю «Дата рождения» — открывается календарь.

### Баг 3: После прохождения демо прогресс показывает 19% вместо 100%

**Симптом:** Кандидат прошёл все блоки демо до конца, дошёл до финальной
страницы — а на странице кандидата (или в списке /hr/candidates) прогресс
показывает 19%, не 100%.

**Где искать:**
- API endpoint расчёта прогресса: возможно
  app/api/modules/hr/candidates/progress/route.ts
- Или функция вычисления прогресса в lib/candidate-progress.ts
- Триггер «прохождение завершено»: app/api/public/demo/[token]/answer/route.ts
  или app/api/public/demo/[token]/apply/route.ts

**Что сделать:**
1. Найди формулу расчёта прогресса. Скорее всего:
   progress = answered_questions / total_questions * 100
2. Проверь что total_questions считается правильно — возможно туда попадают
   опциональные блоки которые кандидат не должен проходить
3. Проверь что финальная страница (__complete__ или is_completed=true) ставит
   прогресс = 100% независимо от формулы
4. Если есть поле candidates.is_completed — при true возвращать 100%

**Тест:** Пройди тестовое демо до конца → проверь /hr/candidates → прогресс
должен быть 100%.

### Баг 4: Двойное нажатие «Далее» на /demo/[token] (race condition)

**Симптом:** На демо-странице кнопка «Далее» при быстром двойном клике
отправляет два запроса. Не блокер, но иногда создаёт дубликаты ответов.

**Где искать:**
- components/demo/demo-client.tsx
- Кнопка submitAnswer / handleNext

**Что сделать:**
1. Найди handleNext / handleSubmit функцию
2. Добавь useState const [isSubmitting, setIsSubmitting] = useState(false)
3. В начале handleSubmit: if (isSubmitting) return; setIsSubmitting(true)
4. В finally блоке: setIsSubmitting(false)
5. На кнопке: disabled={isSubmitting}
6. Опционально: показывать loader на кнопке при isSubmitting

**Тест:** На демо быстро дважды кликни «Далее» — должен сработать только
один запрос.
