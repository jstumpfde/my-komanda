## 0. ЧТО ХОЧЕТ ЮРИЙ (его словами)
═══════════════════════════════════════════════════════════════════════
«На hh есть исходящий поиск: настраиваешь фильтры → система ходит, ищет
кандидатов по критериям, сканирует их, и если подходят — предлагает им
откликнуться на нашу вакансию. Хочу то же внутри Company24, но с AI-оценкой:
не звать всех подряд (тратит лимиты/токены), а сначала прогнать через наш
AI-скоринг и звать только лучших».

Текущая жалоба: «фильтр скудный, собирает всё подряд, тратит кучу токенов».
Причина частично UX (расширенный фильтр СВЁРНУТ — надо раскрыть), частично
правда (модуль Фаза 1, приглашение не доделано, авто-режима нет).

## 2. ОГРАНИЧЕНИЯ hh API (заложены, не игнорировать)
═══════════════════════════════════════════════════════════════════════
1. Лимит просмотров: 50 просмотров резюме из поиска/день на менеджера + 500
   суммарных уникальных/день. Полный GET /resumes/{id} тратит лимит, поиск
   GET /resumes (сниппеты) — НЕТ. Поэтому: AI ранжирует по сниппетам, полный
   просмотр — только для топ перед приглашением. Счётчик — hh_resume_view_quota.
2. Приглашения требуют АКТИВНОГО платного доступа к базе резюме hh у клиента.
   Поиск+просмотр открыты, приглашение — нет. Если доступа нет → кнопка
   «Пригласить» неактивна, статус «Доступ к базе резюме hh не активен».

## 3. hh API — КОНКРЕТИКА (сверено)
═══════════════════════════════════════════════════════════════════════
Токен по company_id (OAuth), lib/hh-helpers getValidToken. Прямые вызовы
api.hh.ru (прокси для hh НЕТ). User-Agent: "Company24/1.0 (company24.pro)".

ПОИСК: GET /resumes — параметры (подтверждены по /dictionaries + /languages):
  text, area (ТОЛЬКО числовой id!), experience (noExperience|between1And3|
  between3And6|moreThan6), salary_from/to, period, per_page (cap 100), page,
  employment[] (full|part|project|volunteer|probation),
  schedule[] (fullDay|shift|flexible|remote|flyInFlyOut),
  education_level (secondary|special_secondary|unfinished_higher|higher|bachelor|
    master|candidate|doctor),
  gender (male|female), relocation (living_or_relocation|living|living_but_
    relocation|relocation — ТРЕБУЕТ area),
  order_by (relevance|publication_time|salary_desc|salary_asc),
  label[] (only_with_photo|only_with_salary|...),
  language[] (формат "{id}.{level}", напр eng.b2; id из /languages, level a1..c2/l1),
  age_from/age_to.
  Множественные — повторяющиеся query (qs.append).

ПРОСМОТР (тратит лимит): GET /resumes/{resume_id} — полные данные.

ПРИГЛАШЕНИЕ (требует доступ к базе) — ❗НУЖНО СВЕРИТЬ ПЕРЕД РЕАЛИЗАЦИЕЙ:
  Офиц. дока: github.com/hhru/api → docs/employer_negotiations.md (и docs/resumes).
  Алгоритм по ТЗ: сначала запросить вакансии работодателя, применимые к резюме
  (приходят arguments + resulting_employer_state), затем создать приглашение.
  Пара «вакансия+резюме» = одно приглашение. ПРОЧИТАТЬ ДОКУ, НЕ УГАДЫВАТЬ.
