## 0. TL;DR

Продажи — это **не стройка с нуля**. У нас есть два готовых актива, которые надо
соединить:
1. **HR-движок = рабочий «мозг»** (на проде): AI-скоринг, чат-бот с 4-уровневой
   безопасностью, дожим, cron-автоматизация, календарь.
2. **Модуль продаж = готовый «скелет», но витрина**: ~18 страниц UI на
   demo-данных + booking + sales-таблицы в схеме, без подключения к движку.

**Задача: протянуть провод между мозгом (HR) и скелетом (sales-UI + sales-таблицы)
+ построить 2-3 новых куска (канал доставки сообщений, публичная запись клиента,
источник лидов).**

## 1. Видение и стратегия (со слов Юрия, 07.06.2026)

- **Продажи — ядро продукта**, интереснее HR. За HR сейчас мало кто платит, за
  лиды/новых клиентов — спрос большой. Надежда: запустив продажи, продавать и
  продажи, и HR-модуль.
- **Ключевой инсайт: «поиск клиентов ≈ поиск кандидатов»** — это буквально один
  движок, направленный на другой объект (лид вместо кандидата).
- **Склад и логистику НЕ строим как отдельные вертикали.** Только как функции
  продаж: склад = «наличие товара», календарь = «продажа времени/слотов».
- **Первый ICP (рекомендация координатора, подтвердить у Юрия): «бизнесы,
  продающие время»** — beauty, стоматология, услуги, посуточная аренда. Почему:
  острая боль с лидами И записью разом; платят за заполненный календарь (ясный
  ROI); меньше конкуренции, чем в товарке; календарь уже частично есть.
- Глубокая вертикаль > размазанная горизонталь. В мире дешёвых ИИ-агентов
  ценность платформы = интеграции + данные + готовый результат под ключ +
  безопасность/ответственность, а НЕ «конструктор с ручной настройкой».
  Стратегический разворот: от «настрой сам» к «результат под ключ, ИИ-оператор
  внутри».

## 4. Архитектурные решения

1. **Переиспользовать ДВИЖОК (логику), а не таблицу `candidates`.** Данные
   продаж — на отдельных sales-таблицах (уже есть). Чище и обходит баг B9.
2. **Баг B9 (две системы статусов кандидата) в новый модуль НЕ тащить.** В
   продажах сразу одна система стадий сделки/лида.
3. **Главный технический пробел — КАНАЛ ДОСТАВКИ.** В HR всё идёт через
   hh.ru-чат (`scan-incoming`, `process-queue`). В продажах hh нет — нужен слой
   отправки/приёма сообщений: email / SMS / Telegram / виджет на сайте. Без него
   дожим и чат-бот некуда подключить.
4. **Чат-бот: ядро (pre-filter, abuse, квота, генератор, watcher) ~100%
   переносимо.** Адаптировать надо: intent-классификатор (salary→budget,
   interview_scheduling→deal_scheduling и т.п.), trigger-систему, примеры в
   SAFETY_RULES/post-filter (оффер→договор, интервью→встреча).
5. **Общее ядро — общий код.** funnel-builder / ai-chatbot / scoring должны стать
   общими для «кандидат | лид», а не копипастой. Иначе два расходящихся дерева
   (как боль B5/B9).

## 5. Первый сквозной сценарий (предложение)

**Beauty/стоматология:** входящий лид с виджета на сайте → AI квалифицирует в
чате (переиспользуем chatbot-processor + промпт под услугу) → записывает в
свободный слот (booking) → дожимает, если не дошёл (follow-up). Даёт законченную
ценность «заполнили календарь» и задействует почти всё переиспользование разом.

## 6. Открытые вопросы для Юрия (СВЕРИТЬСЯ до старта)

1. **Источники лидов — «будет много каналов», Юрий расскажет детали отдельно.**
   Это определяет всю lead-gen часть. Узнать: какие каналы (виджет на сайте /
   реклама / импорт базы / маркетплейсы / соцсети / звонки), какой первый.
2. **Подтвердить первый ICP** (предложено: продажи времени — beauty/стоматология/

3. **Канал доставки сообщений** — с чего начать (Telegram? email? виджет?).
4. **Где работает чат продаж** — Юрий рассматривал: отдельный аккаунт /
   параллельный воркстrim / отдельный прод, чтобы ускорить и не мешать.

## 8. Рабочие правила (обязательно, см. CLAUDE.md + memory)

- **Деплой-поток: сначала стейджинг, потом прод** (develop → new.company24.pro →
  тест → main → прод). НЕ гнать сразу на прод.
- **Деплой только через safe-скрипт, в фоне** (иначе осиротевший build = 502).
- **Дал добро — не переспрашивать**, действовать до конца.
- **Координатор управляет параллелизмом**: фоновые агенты на непересекающихся
  зонах; деплой централизован; агенты не пушат в main.
- Стиль: всё на русском, shadcn/ui + Tailwind, без новых npm-пакетов без
  разрешения, дизайн-эталон — /hr/calendar.
- ORM Drizzle, схема `lib/db/schema.ts` — источник правды; перед API проверять
  колонки.

## 9. Риск координации с основным чатом

Продажи будут переиспользовать **общее ядро** (funnel-builder / ai-chatbot /
scoring), которое сейчас живёт в HR-контексте. Если оба чата одновременно
рефакторят ядро — конфликты. Договорённость:
- Чат продаж работает в своей зоне: `app/(modules)/sales/*`, `app/(modules)/
  booking/*`, `app/(public)/schedule|booking/*`, `app/api/modules/sales|booking/*`,
  новые `lib/sales/*`.
- Изменения в ОБЩЕМ ядре (funnel-builder, ai/chatbot-*, scoring) — согласовывать
  с основным чатом-координатором (вынос ядра в generic делать аккуратно, чтобы не
  сломать живой HR на проде).
