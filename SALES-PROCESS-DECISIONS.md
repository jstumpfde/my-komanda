## A. Принципы и зафиксированное

**Принцип «настраивается = настройка + умный дефолт».** Где Юрий говорит «должно
настраиваться» — делаю per-tenant настройку (на уровне салона/воронки), НО с
разумным дефолтом, чтобы работало «под ключ» сразу, без ручной настройки. Дефолт =
моя рекомендация. Сходится со стратегией «результат под ключ, ИИ-оператор внутри».

**Зафиксировано ранее:**
- **ICP:** продажи времени (beauty / стоматология / услуги / аренда).
- **Архитектура:** модульный монолит — не форкаем. Модули по лицензиям/флагам.
- **Каналы:** все, через pluggable-адаптеры. **Первым — Telegram.**
- **Источники лидов:** все, через адаптеры.
- **Разработка:** своя ветка + тест-тенант на стейджинге. Отдельный прод не форкаем.
- **Рабочий режим:** Claude строит сам; к Юрию — только процессные вопросы.

### Этап 3. Запись в слот
- **3.1** 🔴 Бот сам бронирует или человек подтверждает? → **РЕШЕНО: администратор
  подтверждает.** Бот подбирает и держит слот → статус брони «ожидает подтверждения»
  → салон подтверждает (уведомление). Нужно: статус booking «pending», экран/действие
  подтверждения, уведомление салону.
- **3.2** 🔴 Как предлагаем время? → **РЕШЕНО: настраивается.** Дефолт = гибрид
  (спросить предпочтение → предложить 2-3 конкретных слота). Настройка режима:
  гибрид / сразу ближайшие / по запросу клиента.
- **3.3** 🔴 Откуда услуги/мастера/график? → **РЕШЕНО: гибрид** — даём шаблон
  справочников по нише (beauty/стома), салон правит под себя. Бот работает поверх.
- **3.4** 🟡 Если слотов нет? → 💡 предложить ближайшие (др. мастер/день) → лист
  ожидания + дожим при освобождении. **Решение:**

## C. Журнал решений (8 🔴 — каркас MVP, утверждено 07.06.2026)

1. **Вход лида (0.1):** оба варианта — клиент сам пишет боту И мы пишем по заявке.
   *Очередь: сначала «клиент сам пишет», потом адаптер заявки.*
2. **Первый контакт (1.1):** бот приветствует сразу + вопрос-зацепка.
3. **Квалификация (2.1):** расширенный сбор — услуга, время, мастер(опц.), телефон,
   новый/повторный, пожелания/повод.
4. **Бронирование (3.1):** бот подбирает и держит слот, финально подтверждает
   администратор. → статус брони «pending» + подтверждение + уведомление салону.
5. **Подбор времени (3.2):** НАСТРАИВАЕТСЯ. Дефолт — гибрид (предпочтение → 2-3 слота).
6. **Справочники (3.3):** гибрид — шаблон по нише + правки салона.
7. **Дожим (5.1):** НАСТРАИВАЕТСЯ. Дефолт — 1-2 ч / след. день / макс 2-3, потом стоп.
8. **Эскалация (6.1):** НАСТРАИВАЕТСЯ. Дефолт — по триггерам + по явной просьбе.

**Следствие для архитектуры:** нужен блок настроек sales-чатбота на уровне салона
(подбор времени · дожим · эскалация) — по образцу HR `aiChatbotSettings`/
`responseTiming`. Статус booking расширяется на `pending` (ожидает подтверждения).
