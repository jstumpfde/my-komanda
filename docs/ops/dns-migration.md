# Перенос DNS company24.pro на надёжный (российский) DNS

Памятка на будущее. Делать НЕ срочно, в спокойное время. Прод от этого не зависит —
переключение влияет только на то, как резолверы находят IP сервера.

---

## 1. Зачем это вообще

Симптом: периодически «Safari не может найти сервер company24.pro» / «сайт лёг»,
хотя сервер жив.

Причина (диагностировано 16.06.2026):
- Домен на **NS timeweb** (`ns1/ns2.timeweb.ru`, `ns3/ns4.timeweb.org`).
- Один из NS timeweb (`85.193.93.93`) **периодически таймаутит** (не отвечает).
- На домене включён **DNSSEC**, и строгие резолверы (Google 8.8.8.8) при таймауте NS
  отдают `SERVFAIL` → браузер пишет «сервер не найден».
- Браузер кэширует отрицательный ответ → «лежит», пока не сбросишь DNS-кэш.
- Низкий TTL (600 = 10 мин) учащает перепроверки → больше окон для сбоя.

Вывод: слабое звено — **NS-инфраструктура timeweb + строгий DNSSEC**. Лечится переносом
DNS на более надёжный Anycast-DNS.

## 2. Почему НЕ Cloudflare

Для российской аудитории Cloudflare рискован: РКН периодически душит Cloudflare
(особенно ECH/проксирование). Даже в режиме «DNS only» запросы к Anycast-NS Cloudflare
могут троттлиться из РФ. Берём **российский** провайдер DNS.

Рекомендуемые варианты (бесплатные, РФ):
- **reg.ru DNS** — проще всего, домен уже зарегистрирован в reg.ru, всё в одном кабинете.
- **Selectel DNS** — бесплатный Anycast, обычно стабильнее. Чуть больше телодвижений.

Почта (mx1/mx2.timeweb.ru) и сервер (5.42.125.91, 72.56.241.159) остаются на timeweb —
меняем ТОЛЬКО кто отвечает на DNS-запросы.

> ⚠️ `radar` и `marketradar` смотрят на ОТДЕЛЬНЫЙ IP `72.56.241.159`. Их обязательно

### TXT-записи
| Хост | Значение |
|---|---|
| `@` | `v=spf1 include:_spf.timeweb.ru ~all` |
| `@` | `_globalsign-domain-verification=MJKuFNN8D8vz7JaEaDWPGIm1KISsSfR6BGqtYXx6AB` |
| `_dmarc` | `v=DMARC1;p=none` |
| `dkim._domainkey` | `k=rsa;p=MIIBIjANBgkqhkiG9w0BAQEF…` ⚠️ скопировать ПОЛНОСТЬЮ из панели timeweb |
| `*` (wildcard) | `v=spf1 include:_spf.timeweb.ru ~all` |

> ⚠️ **DKIM** (`dkim._domainkey`) — критично для доставляемости почты. Значение длинное
> (RSA-ключ), в списке timeweb обрезано. Открой запись (⋯ → редактировать/копировать) и
> перенеси значение ЦЕЛИКОМ, без пробелов/переносов.

### НЕ переносить
- `TXT _acme-challenge.*` — временные токены подтверждения SSL (Let's Encrypt/GlobalSign),
  создаются заново при следующем выпуске сертификата.
- SPF-TXT на отдельных `radar`/`marketradar`/`new`/`staging` — дублируют wildcard `* SPF`,
  можно не переносить (но если перенесёшь — не сломается).
- NS-записи timeweb — НЕ переносить, новые NS даст новый провайдер.

## 4. Пошаговый порядок

### Шаг 0. Сними финальный снимок зоны
В панели timeweb (Домены → DNS зона company24.pro) пробегись по ОБЕИМ страницам и сверь
со списком выше. Особое внимание: DKIM (полное значение) и `radar`/`marketradar`.

### Шаг 1. Заведи зону у нового провайдера
**reg.ru:** Личный кабинет → домен company24.pro → «DNS-серверы и управление зоной» →
включить управление зоной на NS reg.ru (`ns1.reg.ru`, `ns2.reg.ru`).
**Selectel:** Панель → DNS-хостинг → создать зону `company24.pro`. Selectel даст свои NS.

### Шаг 2. Создай все записи из раздела 3
Внимательно: `radar`/`marketradar` → 72.56.241.159, DKIM — полностью.

### Шаг 3. ⚠️ Сними DNSSEC на reg.ru (ДО смены NS)
reg.ru → домен → раздел **DNSSEC** → удалить DS-запись / отключить DNSSEC.
Подожди ~1 час, чтобы старый DS «рассосался» из кэшей.

> Если сменить NS, пока старый DS (ключи timeweb) ещё опубликован — домен сломается
> ВЕЗДЕ (несовпадение ключей DNSSEC = SERVFAIL у всех валидирующих резолверов).

### Шаг 4. Смени NS домена
reg.ru → домен → **DNS-серверы (NS)**:
- reg.ru DNS: обычно достаточно включить «использовать DNS reg.ru».
- Selectel: впиши NS от Selectel вместо timeweb.

### Шаг 5. Подожди пропагацию
Обычно 30 мин – 2 часа, иногда до суток.

### Шаг 6. Проверь (терминал)
```
dig company24.pro @8.8.8.8 +short              # → 5.42.125.91
dig company24.pro @1.1.1.1 +short              # → 5.42.125.91
dig radar.company24.pro +short                 # → 72.56.241.159
dig new.company24.pro +short                   # → 5.42.125.91
dig MX company24.pro +short                    # → 10 mx1.timeweb.ru / 20 mx2.timeweb.ru
dig TXT company24.pro +short                   # → SPF + globalsign
dig dkim._domainkey.company24.pro TXT +short   # → k=rsa;p=…
dig NS company24.pro +short                    # → новые NS (reg.ru/selectel)
```
Прогони несколько раз — ответы должны быть стабильными, без пустых.

### Шаг 7. (Опционально) включи DNSSEC на новом провайдере
Включить DNSSEC у нового DNS → он даст новую DS-запись → добавить её в reg.ru → DNSSEC.
Делать только ПОСЛЕ того, как новые NS активны и резолвят.

---

## 5. Быстрый временный костыль (если переносить лень)
Не убирает корень, но снижает частоту «лежит»:
1. На reg.ru **отключить DNSSEC** (удалить DS) — таймаут NS перестанет превращаться в
   SERVFAIL у строгих резолверов.
2. В timeweb поднять **TTL** A-записей с 600 до **3600** — реже перепроверки.

## 6. Откат
Если что-то пошло не так — вернуть NS домена на timeweb
(`ns1.timeweb.ru, ns2.timeweb.ru, ns3.timeweb.org, ns4.timeweb.org`) в reg.ru и
(если включал) убрать новый DS. Через пропагацию вернётся как было.
