# Как работает платформа Company24.pro — поток найма по шагам

> Документ описывает, что происходит на платформе с момента публикации вакансии
> до выхода кандидата на работу: какие функции и крон-процессы задействованы на
> каждом шаге, откуда что берётся. Составлено по фактическому коду (июнь 2026).
>
> Среда: прод company24.pro (ветка develop). Все автоматические процессы —
> крон-эндпоинты `/api/cron/*`, запускаются с сервера по расписанию (crontab),
> авторизация заголовком `X-Cron-Secret`. Логи каждого запуска — таблица `cron_runs`.

## КАРТА ВСЕГО ПОТОКА (одним взглядом)

```
   HR                          ПЛАТФОРМА (авто)                      КАНДИДАТ
   │                                                                    │
 1.│ создаёт вакансию ──┐                                               │
   │                    ├─ анкета + AI-профиль (требования к скорингу)  │
 2.│ заполняет анкету ──┘  AI-ассистент оценивает качество анкеты       │
   │                                                                    │
 3.│ публикует на hh.ru ──────────────────────────────────────────────→│ видит вакансию
   │                                                                    │
   │                                                                    │ 4. ОТКЛИКАЕТСЯ
   │                          ┌─ cron hh-import (~2 мин) ←──────────────┘
   │                          │   тянет отклик, создаёт кандидата (stage=new)
   │                          │
   │                          ├─ ОБРАБОТКА ОТКЛИКА (process-queue):
   │                          │   a) стоп-факторы (жёсткий фильтр)
   │                          │   b) AI-скоринг резюме (resume_score)
   │                          │   c) решение по баллу: демо / предкв / ручной разбор / отказ
   │                          │   d) первое сообщение + демо-ссылка ───────→│ получает приглашение
   │                          │   e) расписывает дожимы (follow-up)         │
   │                          │                                            │ 5. проходит ДЕМО
   │                          ├─ cron hh-incoming (~15 мин):               │    отвечает в чате
   │                          │   AI чат-бот ведёт диалог ←────────────────┤
   │                          │   (отвечает / эскалирует / отказывает)     │
   │                          │                                            │ 6. заполняет АНКЕТУ
   │                          ├─ AI-скоринг анкеты (ai_score)              │    проходит ТЕСТ
   │                          ├─ cron follow-up (~1 мин): дожимы молчунам  │
   │                          ├─ cron pending-rejections (~5 мин):        │
   │                          │   отложенные отказы в рабочее время        │
   │                          │                                            │
 7.│ смотрит воронку, ←───────┴─ кандидаты разложены по стадиям            │
   │ приглашает/отказывает                                                │ 8. ВЫХОД на работу
   ▼                                                                      ▼
```

## ШАГ 1-2. ПОДГОТОВКА ВАКАНСИИ (делает HR, до публикации)

**Что делает HR:** создаёт вакансию, заполняет анкету (вкладка «Анкета»).

**Что задействовано на платформе:**
- **9 блоков анкеты** (`components/vacancies/anketa-tab.tsx`): должность, мотивация
  (ЗП/бонусы), о компании, обязанности+требования, портрет кандидата (навыки),
  условия, документы, AI-генерация, **AI-профиль кандидата**.
- **AI-ассистент** (правая панель, `vacancy-advisor.tsx` + `/api/ai/vacancy-advisor`):
  в реальном времени оценивает качество анкеты (% заполнения, баллы названия,
  аналитика зарплат vs рынок, рекомендации, стоп-факторы). Кэшируется; чтобы
  пересчитать — кнопка «Переанализировать».
- **AI-профиль кандидата** (блок 9) — это ТРЕБОВАНИЯ для будущего скоринга:
  - «Идеальный кандидат» (свободный текст) — AI сравнивает с ним каждое резюме
  - «Обязательные компетенции» (hard skills) — штраф по доле несовпадения
  - «Стоп-факторы» — жёсткий отсев (рейтинг 0)
  - «Минимальный опыт», «Критерии оценки с весами» (критично/важно/желательно)
  - «Вопросы для скрининга» — учитываются AI при оценке резюме
- **Откуда дефолты:** при создании вакансии копируются дефолты компании
  (`/hr/hiring-settings`): расписание, стоп-факторы (если включён флаг).

**Важно:** анкета — это «мозг» вакансии. Чем точнее AI-профиль, тем точнее
скоринг откликов на шаге 4.

## ШАГ 3. ПУБЛИКАЦИЯ НА HH.RU

**Что делает HR:** связывает вакансию с hh.ru (вкладка «Настройки» → Интеграции)
или публикует напрямую.

**Что задействовано:**
- OAuth-токен компании (`hh_integrations`, по `company_id`). Авто-рефреш
  истёкших — `lib/hh-helpers.ts getValidToken`.
- Cron `hh-token-refresh` — обновляет токены (не трогает не истёкшие — hh
  ругается «token not expired»).
- `vacancies.hhVacancyId` / `hhUrl` — связка с hh-вакансией.

После публикации кандидаты видят вакансию на hh.ru.

## ШАГ 4. КАНДИДАТ ОТКЛИКНУЛСЯ → ПРИЁМ И ОБРАБОТКА ОТКЛИКА

Это сердце платформы. Происходит автоматически, в несколько под-шагов.

### 4.0. Импорт отклика — cron `hh-import` (каждые ~2 минуты)
Файл: `app/api/cron/hh-import/route.ts` → `lib/hh/process-queue.ts`
- Тянет новые negotiations с hh по всем активным вакансиям.
- Лимит 50 откликов на компанию за вызов (rate-limit hh ≈ 60 rps, обработка
  последовательная с задержкой 2 сек/отклик).
- Защита от параллельных запусков: `pg_advisory_lock` (если предыдущий cron
  ещё работает → 409, не дублирует).
- Создаёт запись `candidates` (source=hh, stage=**new**), тянет поля резюме
  (город, ЗП, опыт, навыки, образование, фото).

### 4.a. Стоп-факторы — ЖЁСТКИЙ ФИЛЬТР (до скоринга)
Файл: `lib/funnel-builder/stop-factors-matcher.ts`
- Проверяются ДО AI-скоринга (экономит токены): город / формат / возраст /
  опыт / гражданство / зарплатные ожидания (`vacancy.stopFactorsJson`).
- Если сработал — кандидат идёт в ветку отказа (см. 4.f), скоринг не запускается.
- Ночью (вне рабочего времени вакансии) стоп-факторы НЕ применяются
  (off-hours soft mode — никаких авто-отказов ночью).

### 4.b. AI-скоринг резюме — `resume_score` (выставляется ОДИН раз)
Файл: `lib/ai-screen-resume.ts` (модель Haiku — дёшево/быстро)
- Сравнивает резюме с AI-профилем вакансии (идеальный кандидат, hard skills,
  стоп-факторы, вопросы скрининга).
- Выдаёт 0-100 + вердикт (match / weak / stop) + краткое summary.
- Если уже оценён (resume_score не пуст) — пропускается, токены не тратятся.

### 4.c. Решение по баллу (два порога)
- `score ≥ верхний порог` → обычное приглашение (демо).
- `score < нижний порог` → мягкий отказ (ветка 4.f).
- между порогами → по настройке вакансии:
  - `keep_new` — оставить «новым», ждёт ручного разбора HR,
  - `direct_demo` — пригласить на демо,
  - `prequalification` — запустить опросник предквалификации
    (`lib/prequalification/start.ts`), если вопросы заданы.
- ⚠️ Авто-отказ по баллу сейчас ОТКЛЮЧЕН (P0-14) — между порогами отказа нет,
  только демо / ручной разбор / опросник.

### 4.d. Первое сообщение + демо-ссылка
- Если кандидат прошёл — платформа шлёт в hh-чат первое сообщение с ссылкой
  на демо (`first_messages_chain` — серия до 3 сообщений с задержками; или
  off-hours-вариант для нерабочего времени).
- Стадия → **primary_contact**, hh-отклик помечается `invited`.

### 4.e. Расписание дожимов (follow-up)
- Платформа расписывает цепочку касаний `follow_up_messages` — если кандидат
  молчит, ему напомнят (см. cron follow-up, шаг 6).

### 4.f. Ветка ОТКАЗА (стоп-фактор / низкий балл / провал предкв)
Файл: `lib/rejection/execute.ts`
- **Заход 3 (новое):** отказы НЕ мгновенные. Кандидат помечается «ожидает
  отказа» (`pending_rejection_at` = сейчас + задержка вакансии, дефолт 5 ч),
  остаётся в текущей стадии.
- Исполняет cron `pending-rejections` (каждые 5 мин) — только в рабочее время
  вакансии, шлёт сообщение об отказе в hh + discard, ставит stage=**rejected**.
- Стоп-факторный текст отказа сохраняется на момент планирования (не теряется).

## ШАГ 5. КАНДИДАТ ПРОХОДИТ ДЕМО + ОБЩАЕТСЯ В ЧАТЕ

**Что делает кандидат:** открывает демо-ссылку (`/demo/[token]`), смотрит
демонстрацию должности, отвечает в hh-чате.

**Что задействовано:**

### AI чат-бот — общается с кандидатом в hh-чате
Cron `hh-incoming-messages` (каждые ~15 минут), ядро
`lib/ai/chatbot-processor.ts processChatbotMessage`.
4-уровневая безопасность:
1. **Pre-filter (Haiku)** — проверяет входящее сообщение кандидата
   (injection / мат / код) ДО основного бота.
2. **Executor (Sonnet)** — главный респондер, отвечает по
   `vacancy.aiChatbotPrompt`.
3. **Post-filter (Haiku)** — проверяет ответ бота ДО отправки (запрещённые
   обещания / утечка систем-промпта / уход от темы).
4. **AI Watcher** — периодический аудит последних сообщений (cron
   `ai-chatbot-watcher`, раз в час, cooldown 30 мин).

Что делает бот по итогу разбора сообщения:
- `sent` — отвечает кандидату;
- `rejected` — кандидат сам отказался / стоп-слово / security → отказ
  (для разговорных точек отказ пока мгновенный, см. «незакрытые хвосты»);
- `escalated` — зовёт HR (нестандартная ситуация).

Стадия по ходу демо: **demo_opened** (открыл демо).

Kill-switch бота: `vacancy.aiChatbotEnabled` / `companies.aiChatbotKilled` /
платформенный emergency (`/admin/platform`).

## ШАГ 6. АНКЕТА КАНДИДАТА + ТЕСТ + ДОЖИМЫ

**Что делает кандидат:** заполняет анкету (отвечает на вопросы), проходит
тестовое задание (`/test/[token]`).

**Что задействовано:**
- **AI-скоринг анкеты** — `ai_score` (`lib/ai-screen-candidate.ts`, Sonnet) —
  после демо, учитывает ответы кандидата. Это уже более глубокая оценка, чем
  resume_score на шаге 4.
- **Рубрика (shadow)** — `rubric_score` (`lib/scoring/rubric.ts`, Sonnet,
  forced tool-use) — «правильная» система: оценивает каждый критерий + цитата
  из резюме + уверенность, итог считает КОД взвешенной суммой. Cron
  `rubric-score` доскоривает (берёт `rubric_score IS NULL`).
- Стадии: **anketa_filled** → **test_task_sent** → **test_task_done** →
  **test_passed** / **test_failed**.

### Дожимы — cron `follow-up` (каждую ~1 минуту)
- Шлёт очередное касание молчащим кандидатам из `follow_up_messages` в hh-чат.
- Останавливается, если кандидат ответил / вакансия закрыта.
- Есть «аварийное повторное сообщение» (`recovery_message`).

### Предквалификация — cron `prequalification` (каждые ~15 мин)
- Если включён опросник: задаёт вопросы, при провале критичных — отказ
  (через отложенный механизм), при прохождении — демо.

## ШАГ 7. HR РАБОТАЕТ С ВОРОНКОЙ

**Что видит HR (вкладка «Кандидаты»):**
- Серверная пагинация, фильтры, сортировка, bulk-действия.
- Колонки: AI-резюме (resume_score), AI-оценка (ai_score), рубрика, стадия и др.
- Карточка кандидата (drawer): контакты, ответы, hh-чат, AI-оценка, рубрика,
  история стадий.
- Инструменты: «AI-оценить новых», «Сравнить топ», «Оценить рубрикой».
- Действия по кандидату: Пригласить / Отказать / Открыть.

**Канон стадий воронки** (`lib/stages.ts`):
```
new → primary_contact → demo_opened → anketa_filled →
test_task_sent → test_task_done → test_passed →
interview → offer_sent → hired
                                  ↘ rejected (терминальная)
                                  ↘ test_failed
```

**Аналитика** (вкладка «Аналитика»): воронка, источники, распределение баллов
(серверная агрегация).

**Исходящий подбор** (вкладка «Исходящий подбор»): HR сам ищет резюме на hh
(фильтр как на hh) → AI-скоринг по сниппетам → приглашение. Отдельный поток,
не зависит от входящих откликов.

---

## ШАГ 8. РЕШЕНИЕ И ВЫХОД

- HR проводит интервью (**interview**), отправляет оффер (**offer_sent**,
  AI-генерация оффера), кандидат выходит на работу (**hired**).
- Дальше — модуль адаптации (онбординг, бадди, обучение) — отдельный поток.

## КРОН-ПРОЦЕССЫ (что крутится в фоне, сводка)

| Cron | Частота | Что делает |
|---|---|---|
| `hh-import` | ~2 мин | тянет отклики, запускает обработку (шаг 4) |
| `hh-incoming-messages` | ~15 мин | входящие сообщения → AI чат-бот (шаг 5) |
| `follow-up` | ~1 мин | дожимы молчащим кандидатам |
| `prequalification` | ~15 мин | опросник предквалификации |
| `pending-rejections` | ~5 мин | отложенные отказы в рабочее время |
| `rubric-score` | период | доскоринг рубрикой |
| `ai-chatbot-watcher` | ~1 час | аудит диалогов бота (cooldown 30 мин) |
| `hh-token-refresh` | период | обновление истёкших hh-токенов |
| `trash-cleanup` | раз в сутки 03:00 МСК | авто-удаление корзины старше N дней |

Авторизация всех кронов — заголовок `X-Cron-Secret` (env `CRON_SECRET`).
Каждый запуск логируется в `cron_runs`.

## КЛЮЧЕВЫЕ ТАБЛИЦЫ БД (источник правды — lib/db/schema.ts)

- `companies` — тенант (hh-токены, дефолты найма, брендинг)
- `vacancies` — вакансии (`descriptionJson.anketa` = вся анкета + AI-профиль;
  `stopFactorsJson`; `aiChatbot*`; расписание; `first_messages_chain`)
- `candidates` — кандидаты (stage, source, resumeScore/aiScore/rubricScore,
  hh-данные, `pending_rejection_*`)
- `hh_responses` — сырые отклики с hh (очередь на разбор)
- `ai_chatbot_messages` — журнал диалога бота
- `demos` / `test_submissions` — демо и тесты
- `follow_up_messages` / `follow_up_campaigns` — дожимы
- `cron_runs` — журнал запусков крон-процессов

## ТРИ СИСТЕМЫ AI-СКОРИНГА (не путать)

| Система | Когда | Движок | Что оценивает |
|---|---|---|---|
| **resume_score** | при приёме отклика (до демо) | Haiku | резюме vs AI-профиль |
| **ai_score** | после демо | Sonnet | резюме + ответы анкеты |
| **rubric_score** | shadow / по запросу | Sonnet | покритериально, итог считает код |

Все три берут требования из анкеты вакансии (блок «AI-профиль кандидата»).

## НЕЗАКРЫТЫЕ ХВОСТЫ (честно, на июнь 2026)

- **Отложенные отказы (Заход 3):** переключены 2 системные точки (стоп-факторы,
  провал предкв). Разговорные точки чат-бота (кандидат сам отказался, security)
  пока отказывают мгновенно — требуют решения по поведению.
- **Исходящий подбор «Пригласить»:** поиск+скоринг работают; формат hh
  negotiations сверён с докой, но боевой тест на реальном резюме не делался.
- **Telegram-бот для кандидатов:** только фундамент данных, не построен.
- **Рыночный бенчмарк времени публикации:** карточка «Лучшее время публикации»
  считает по откликам своей компании, не по рынку hh (у hh нет такого API).
