# HANDOFF — контекст для нового аккаунта/помощника

Этот файл — снимок того, что раньше жило только в приватной памяти Claude на
Mac Юрия (`~/.claude/projects/.../memory/`), теперь закоммичен в репозиторий,
чтобы **любой аккаунт Claude Code**, открытый в этой папке, сразу получил
контекст без доступа к чужой локальной памяти.

Порядок чтения для новой сессии: сначала `CLAUDE.md` (правила, стек, критика),
потом этот файл (рабочий стиль + снимок состояния), потом `git log --oneline -20`
и `docs/architecture/` — за актуальным техническим контекстом.

**Важно:** раздел «Снимок состояния» ниже — это факты на дату коммита этого
файла (см. дату в git blame/git log), не источник истины на будущее. Для
актуального состояния всегда доверяй `git log`, коду и `psql`, а не этому файлу.

## Как работать с Юрием (стабильные правила, не протухают)

- **Автономность**: не ждать, не переспрашивать, если можешь решить сам. Дал
  варианты и тишина ~5 минут → выбери лучший по своему анализу, сделай,
  отчитайся ретроспективно. Вопрос — только на настоящей развилке (деньги,
  необратимость, вкусовое продуктовое решение); и даже тогда — предложи дефолт
  и продолжай параллельно остальные пункты.
- **Задание «на ночь»** = цикл до результата: сделал → проверил → следующий
  пункт очереди, без пауз «жду добро». Один сводный отчёт в конце.
- **Показ результата**: не отправлять «посмотреть» без собственной визуальной
  проверки (браузер/скриншот) — прикладывать доказательство, а не просить
  Юрия проверить руками.
- **Не хардкодить** настраиваемый контент — данные идут платформа → компания →
  вакансия; никаких скрытых зашитых фраз, только видимые редактируемые поля.
- **Health-check после деплоя** — всегда СО СТОРОНЫ СЕРВЕРА или через внешний
  VPS (`ssh riga curl ...`), никогда с Мака напрямую — Мак-curl может врать
  (DNS/VPN-артефакты, минимум один инцидент 09.07 был чисто VPN-проблемой
  пользователя, не сервера).
- **Пароли/секреты** — никогда не вводить самому; авторизованный прод — через
  залогиненный Chrome Юрия, не через логин формы с паролями.
- **Помечать команды** `[Mac]`/`[сервер]`, давать ETA.
- **Деплой** — централизованно через координатора (тот, кто ведёт диалог
  сейчас); агенты/параллельные сессии не пушат в main и не деплоят сами.
- **Модели агентов** (экономия токенов, решение 02.07): главный цикл — модель
  сессии; агенты-исполнители (реализация, разведки, миграции, UI) —
  `model: "sonnet"`; сложное рассуждение/адверс-проверки — `model: "opus"` если
  Sonnet не тянет; тривиальная механика — `model: "haiku"`. Fable — только
  координация, не в субагентах.
- Юрий — мужчина, отвечать на русском.

## Git / деплой — граблы, которые уже дорого стоили

- **НИКОГДА** не пушить в main напрямую, не делать `git pull origin main` без
  проверки `git log origin/main..HEAD` — на проде бывают hotfix-коммиты,
  которых нет в develop.
- **Проверять номер миграции на СВЕЖЕМ `origin/main`/`origin/develop`
  непосредственно перед мерджем**, не только перед началом работы — параллельная
  ветка может занять тот же номер, пока ты работал (два реальных инцидента:
  0187 и дважды 0269→0270 в июле 2026; стандартный деплой
  `psql -f drizzle/NNNN_*.sql` при коллизии номера либо роняет вторую
  миграцию, либо падает).
- **Перед завершением git pull/merge на сервере** — проверять нет ли на
  origin/main НОВЫХ коммитов от кого-то ещё, которые появились уже ПОСЛЕ
  твоего пуша (пока ты собирал шедоу-билд) — если появились, нужно
  пересобрать шедоу с актуального HEAD, а не тихо задеплоить частичный срез
  (иначе получится: миграция другой фичи уже накатана на прод-БД, а код для
  неё не задеплоен — путаное полусостояние).
- **Рабочее дерево координатора** (`/Users/juri/Projects/my-komanda`, обычно
  ветка `tmp/batch-pending`) почти всегда СТАРЕЕ origin/main и часто грязное
  (чужие uncommitted изменения) — анализ и любая новая фича делаются в СВЕЖЕМ
  git worktree от `origin/main`, никогда не в этой папке напрямую.
- **Blue-green деплой** (стандарт с 02.07) — рецепт по шагам в скилле
  `company24-deploy`. Zero-downtime, проверено много раз. Fallback —
  `/root/deploy-prod-safe.sh` в фоне с поллингом (даёт секунды простоя на pm2
  reload).
- **`pm2 reload`, никогда `restart`** на живом проде (restart = 5-8 сек
  простоя). **Никогда `rm -rf .next`** на живом проде в рабочие часы.
- **predeploy-guard** — обязателен перед каждым нетривиальным прод-деплоем.
  Если ревьюер получает контекст только через args-строку и не видит нужную
  ветку/worktree явно — он может ошибочно проверить рабочее дерево
  координатора вместо реальной фичи и выдать ложный STOP. Явно указывать
  `git diff origin/main origin/develop` и конкретные коммиты в args, если
  фича живёт не в текущей папке.
- **Общий worktree — никогда `git add -A`**, только конкретные файлы.

## Архитектура и домен (коротко — подробности в docs/architecture/ и CLAUDE.md)

- **Стек**: Next.js (App Router), TypeScript, Drizzle ORM (НЕ Prisma), Postgres,
  pnpm, shadcn/ui + Tailwind.
- **Модульный монолит** — одно ядро + модули по флагам (`(modules)` route
  group), не форкать под новых клиентов. Модуль включается компании через
  `companies.enabled_modules` (grandfather-режим: null = показывать по роли).
- **Роли**: `platform_admin`/`admin` (роль) ≠ владелец платформы. Реальный
  владелец обычно `role="director"` СВОЕЙ ЛИЧНОЙ компании, платформенный
  статус даёт ТОЛЬКО email из `PLATFORM_ADMIN_EMAILS`
  (`session.user.isPlatformAdmin`). Гейтить owner-only функциональность нужно
  этим полем, а не ролью — путаница уже ловилась минимум дважды (клиентские
  модули 03.07, Big Life 09.07).
- **Партнёрская impersonation** («Войти как клиент») прозрачно свапает
  `companyId`/`effectiveRole` в сессии — код, написанный через
  `requireCompany()`/`requireDirector()`, автоматически работает и для
  партнёра под impersonation, без доп. усилий.
- **Стоп-факторы, Портрет-скоринг, Воронка v2** — три разные скоринговые
  системы с разной ролью в пайплайне (см. memory/scoring-three-systems и
  memory/funnel-v2-*, если доступна, иначе docs/architecture/).
- **Big Life** (biglife.company24.pro) — статический сайт-витрина (НЕ Next.js,
  чистая nginx-статика на том же сервере, `/var/www/client-sites/biglife`, НЕ
  `client-pages` — устаревший путь). С 09.07 заведён как обычный тенант
  платформы (компания + director-аккаунт), управление обложками через
  `/big-life/covers` (обычная company-scoped страница, НЕ platform-admin).
  «Опубликовать» пишет HTML напрямую на диск сервера (fs, без rsync — оба
  процесса на одной машине).

## Как дать доступ другому аккаунту/помощнику

1. **GitHub** — добавить как collaborator в приватный репозиторий (Settings →
   Collaborators), тогда их Claude Code сможет клонировать/пуллить код и
   сразу получит `CLAUDE.md` + этот файл.
2. **Claude-аккаунт** — у каждого помощника нужен свой логин (общих логинов
   нет). Team/Enterprise-план — приглашение сиденьями от админа; личный
   Pro/Max — отдельная подписка на человека.
3. **Сервер (SSH)** — отдельное решение о доверии. Если помощник только пишет
   код (не деплоит) — сервер не нужен вообще: работа в изолированном worktree,
   пуш в ветку, деплой остаётся централизованным через координатора (см.
   правило выше про «агенты не деплоят»).
4. Локальная память Claude (не этот файл) не переносится между аккаунтами —
   если что-то из рабочего стиля оказалось важным для следующей сессии,
   дописывай в этот файл или в `CLAUDE.md`, а не полагайся на то, что «Claude
   и так запомнит».
