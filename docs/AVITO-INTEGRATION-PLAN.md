# Avito-интеграция — план и архитектура (08.06.2026)

> Avito нужен и HR (отклики/чаты по вакансиям), и продажам (чаты по товарам/услугам).
> Канальный слой `lib/channels/*` общий с sales-чатом — координируем, НЕ дублируем.
> Готовим ВЫКЛЮЧЕННЫМ (feature-flag off), включается, когда клиент подключит Avito.

## Главный принцип: один движок + слой политик канала (compliance-by-design)

Не дублируем конфиг воронки на каждый канал. Вместо этого:

1. **Источник.** Вакансия (и sales-объект) имеет набор источников: `hh` | `avito` | оба.
   Входящие с любого источника → ОДИН общий pipeline. `candidate.source` тегирует
   происхождение (колонка «Источник» уже есть).
2. **Конфиг воронки — один, канально-агностичный** (сообщения/стадии/дожим/чат-бот).
3. **CHANNEL_POLICY[channel]** — тонкий дескриптор политики канала. Единственный
   источник правды о том, что канал запрещает/трансформирует. Работает в ДВУХ местах:
   - **UI:** при настройке воронки, если активен источник Avito — поля, нарушающие
     политику, дизейблятся/валидируются с подсказкой. Бан-контент настроить НЕЛЬЗЯ.
   - **Рантайм:** адаптер канала прогоняет каждое исходящее сообщение/действие через
     политику — режет/блокирует запрещённое (defense-in-depth, даже если просочилось).
4. **Минимальные per-channel оверрайды** — только где вынужденно (напр. «первое
   сообщение для Avito» без ссылки). Паттерн как у «сообщение для нерабочего времени».
   НЕ копия всего конфига.
5. **Опционально, off по умолчанию.** Feature-flag на компанию + выбор источника на
   вакансии. Клиенты без Avito лишнего не видят.

## Что Avito может банить (заполнить CHANNEL_POLICY[avito]) — ВЕРИФИЦИРОВАТЬ по доке Avito
Черновой список (уточнить на developers.avito.ru / поддержке):
- внешние ссылки и номера телефона в чате/первом сообщении (увод с площадки);
- массовый авто-первый-контакт (нежелательный спам), лимиты на частоту авто-сообщений;
- агрессивный увод в другие мессенджеры;
- запрещённые категории/контент.
Политика: `forbidExternalLinks`, `forbidPhoneInChat`, `forbidAutoFirstContactBroadcast`,
`maxAutoMessagesPerDay`, `sanitize(text)` — и т.п.

## Подключение (без дорогого «Максимального»)
- Дешевле: **«Расширенный»** тариф + **Messenger API** (`messenger:read/write`) — даёт
  переписку. Дорогой **Jobs API** («Максимальный» + pay-per-response) НЕ нужен, если
  достаточно чатов. Проверить, покрывает ли «Расширенный» чаты по ВАКАНСИЯМ.
- OAuth: путь A — ключи клиента (`client_credentials`), без одобрения Avito (MVP);
  путь B — партнёрский `authorization_code` (кнопка «Подключить»), регистрация
  приложения у Avito, позже.
- Разработка/тест — бесплатно: 7 дней + 100 запусков при регистрации приложения.

## Скелет (готовим сейчас, выключено)
- `lib/channels/policy.ts` — CHANNEL_POLICY (hh, avito): capability + запреты + sanitize.
- `lib/channels/avito.ts` — адаптер Messenger API под интерфейс `lib/channels/types.ts`
  (send/receive/webhook), guard: no-op без ключей и без feature-flag.
- Интеграция «Подключить Avito» (Настройки → Интеграции) — ключи/OAuth, off по умолчанию.
- Источник на вакансии (`sources: ['hh'|'avito']`) — выбор площадок.
- Колонки «Действие в Авито/SuperJob» в таблице стадий — УЖЕ есть; связать с политикой.
- Применение политики в UI воронки (дизейбл/валидация) — фаза 2.

## Координация (важно)
Канальный слой `lib/channels/*` активно строит sales-чат (Telegram-адаптер, sales-bot).
Avito-адаптер делать СИБЛИНГОМ под их интерфейс (новые файлы avito.ts/policy.ts —
без конфликтов), интеграционные точки (process-queue, funnel-UI) — HR-сторона.
Согласовать с sales-чатом, чтобы Avito-канал был общим для HR и продаж, не два разных.
