# Агент коммуникаций — дизайн (пилот на дожимах)

Дата: 10.07.2026. Согласовано с Юрием в чате (сжатый цикл — Юрий уходил на
3-4 часа, дизайн подтверждён по переписке, не по формальному ревью файла).

## Проблема

Юрий сформулировал две вещи, слитые в одну задачу:
1. «Сложно синхронизировать» — настройки коммуникаций (первое сообщение,
   дожимы, отказ, off-hours, отмена интервью, ссылка на встречу) разбросаны
   по разным местам вакансии.
2. «Агент подстраивает текст под кандидата в рамках заготовки HR» —
   генеративная адаптация вместо буквальной подстановки {{переменных}}.

## Находка: проблема #1 уже решена

Вкладка «Коммуникации» (`settingsSection === "communications"` в
`app/(modules)/hr/vacancies/[id]/page.tsx:4679`) УЖЕ существует и собирает
все настройки в одном месте: Первый контакт, Дожимы, Диалог, Интервью,
Отказы, Публикация. Там же уже есть комментарий-маркер (`#62`) от более
ранней сессии, предвосхищающий именно эту задачу: «когда обработка
[агентом] заработает — блоки ниже будут заглушены AI-агентом».

Вывод: не нужно ничего консолидировать заново — нужно РЕАЛИЗОВАТЬ то, что
уже анонсировано в UI, но не подключено («Обработка пока не подключена»).
Task #1 в трекере помечен как pending ошибочно — по факту UI готов, только
не хватает механики.

## Решение по проблеме #2 (эта сессия)

**Не начинать с отказов.** У отказов уже есть осознанное решение (см. memory
`legal-rejection-texts-neutral-keep-autoreject`) — текст юридически
нейтральный, это НЕ место для генеративной вариативности (комплаенс-риск
выше выгоды). Начинаем с дожимов (follow-up) — низкая ставка, не юридический
текст, легко откатить.

**Живой AI-чат-бот (`lib/ai/chatbot-processor.ts`, реактивные ответы в
диалоге) НЕ трогаем.** Юрий сам отметил, что его «не можем нормально
запустить» — тащить ту же генеративную неопределённость в ещё больше мест
ДО того, как решена нестабильность оригинала, увеличивает риск, а не
снижает его. Это отдельная, более поздняя задача (диагностировать сначала).

## Архитектура пилота

```
follow_up_messages.messageText (шаблон, как сейчас)
  → renderTemplate(...) → literal finalText   (ТЕКУЩЕЕ поведение, без изменений)
  → [ЕСЛИ vacancy.aiProcessSettings.dozhimAgentEnabled === true]
      → adaptFollowupMessage({ guardrailText: finalText, ... })
      → safe=true  → finalText = адаптированный текст
      → safe=false → finalText остаётся литеральным (фолбэк, нулевой риск)
```

**Файлы:**
- `lib/comms-agent/adapt-followup-message.ts` — ядро: один AI-вызов
  (AI_MODEL_MAIN, `thinking: disabled`, как в `score-answers.ts`), промпт
  явно требует не менять ссылки/факты/тон, только перефразировать под
  контекст (ветка дожима + номер касания).
- Гварды ПОСЛЕ AI-вызова (дёшево, без второго вызова):
  1. Все URL из заготовки должны быть в выводе буквально.
  2. Длина результата в пределах 0.4×–2× от заготовки.
  3. Пустой/упавший AI-вызов → `safe=false`.
- `app/api/cron/follow-up/route.ts` — единственная точка отправки дожимов
  (`finalText = renderTemplate(...)` на строке ~800) — туда добавлен вызов
  агента за флагом. Флаг `dozhimAgentEnabled` — НЕ включён ни у одной
  вакансии (новое опциональное поле в `VacancyAiProcessSettings`,
  `lib/db/schema.ts`).
- `scripts/preview-dozhim-agent.ts` — сухой прогон на реальных pending-
  касаниях вакансии, печатает литеральный и адаптированный текст рядом,
  НИЧЕГО не отправляет и не меняет в БД. Запускать:
  `pnpm exec tsx --env-file=.env --env-file=.env.local scripts/preview-dozhim-agent.ts --vacancy=<uuid>`
  (оба --env-file обязательны — CLAUDE_PROXY_URL лежит в `.env`, без него AI-
  вызов уходит напрямую на api.anthropic.com и ловит 403 с прод-сервера).

## Что проверено 10.07

Прогнан на реальных pending-дожимах вакансии Revoluterra (3 кандидата, ветка
`opened_not_finished`) — все 3 прошли гварды (safe=true), ссылки сохранены
дословно, тон уместный, длина сопоставимая. Примеры — в логе прогона,
покажу Юрию при возвращении.

## Что НЕ сделано (следующие шаги, ждут решения Юрия)

1. **Ни у одной вакансии флаг не включён** — пилот не влияет на живых
   кандидатов, пока Юрий явно не решит его включить (сначала — просмотреть
   примеры, возможно доточить промпт).
2. **Нет UI-тумблера** — включать пока можно только вручную через
   `PUT /vacancies/[id]/ai-settings` (поле не добавлено туда явно — сделать
   вместе с тумблером, когда будет решение включать).
3. **Расширение на остальные типы** (приглашение, отмена интервью, ссылка на
   встречу, второе демо) — тем же паттерном, после подтверждения качества на
   дожимах.
4. **Почему живой чат-бот «не запускается нормально»** — отдельная
   диагностика, не в этой сессии.
5. **UI-маркер `#62`** («блоки будут заглушены агентом») стоит уточнить —
   решение сейчас: агент ДОПОЛНЯЕТ (переписывает) текст, а не глушит блоки
   целиком; блоки/настройки остаются источником заготовок.
