# Дизайн: единая воронка + агент коммуникаций (вариант A)

Дата: 2026-07-13. Статус: утверждён Юрием (координатор-сессия), готов к планированию.

## Проблема

Сегодня (13.07) на проде случился инцидент: у вакансии в `vacancy_specs.spec.resumeThresholds.inviteHhStage` осталось устаревшее значение `"consider"` (от старого именования до фикса 29.06/11.07). Из-за этого 30+ реальных кандидатов получили приглашение, но их заявка на hh.ru попала не в папку «Первичный контакт», а в «Подумать» — HR их не увидел вовремя.

Разбор показал корневую причину шире одного протухшего поля: в проекте одновременно живут **три независимые системы**, определяющие путь кандидата по воронке:

1. **Легаси-пайплайн** (`lib/stages.ts`, `VacancyPipelineV2`, поле `vacancies.description_json.pipeline`) — список стадий вкл/выкл + пресеты (fast/standard/deep). Своего UI для редактирования практически нет («До Ф3 запись/чтение — minimal», по комментарию в коде).
2. **Старый конструктор из 17 блоков** (`lib/funnel-builder/blocks.ts`, вкладка «Воронка») — визуальный, но не привязан к каноническим `StageSlug`.
3. **«Воронка v2»** (`lib/funnel-v2/types.ts`, `components/vacancies/funnel-v2-builder.tsx`, вкладка «Воронка v2») — новый визуальный конструктор с реальным движком выполнения (гейты по баллу, дожимы на каждой стадии, ветки «прошёл/не прошёл»), но пока опционален (тумблер «Движок воронки v2» по умолчанию выключен, легаси-путь — дефолт).

Ни агент коммуникаций, ни HR не могут однозначно сказать «вот тут настраивается воронка» — конфигурация размазана, и именно рассинхрон одного из полей (`inviteHhStage`) сегодня привёл к инциденту с реальными кандидатами.

## Решение

### 1. «Воронка v2» становится единственной системой

Она уже ближе всего к нужному: канонические стадии, гейты по баллу, ветвление, дожимы на стадию. Решение — **не строить новый экран с нуля**, а:
- Сделать «Воронка v2» ОБЯЗАТЕЛЬНОЙ (снять тумблер-опциональность, движок всегда включён для новых вакансий).
- Вывести из эксплуатации старый 17-блочный конструктор («Воронка») и легаси-пайплайн как отдельные UI-поверхности — миграция существующих вакансий на «Воронка v2» (объём миграции — отдельный вопрос для этапа планирования: сколько вакансий сейчас реально сидит на старых системах).
- Первое приглашение кандидата ВСЕГДА идёт в папку hh «Первичный контакт» (`phone_interview`) — жёсткий, недрейфующий дефолт. Конкретно: (а) при сохранении `vacancy_specs.spec.resumeThresholds.inviteHhStage` валидировать значение против списка допустимых `HhAction`, отклоняя/автоисправляя устаревшие значения (`consider` и другие алиасы прошлого именования); (б) разовая миграция данных — почистить уже сохранённые устаревшие значения на всех вакансиях платформы (по образцу сегодняшней ручной находки — 4 вакансии с `consider`).

### 2. Агент коммуникаций — вариант A (адаптирует текст, не решает)

Подтверждено Юрием явно: агент **не** принимает решения о переходах между стадиями и **никогда** не решает вопрос отказа — это исключительно зона детерминированных правил/гейтов «Воронки v2» (баллы, пороги, факт события). Агент работает НАД уже принятыми решениями: адаптирует текст сообщения под контекст конкретного кандидата в рамках заготовки (см. уже работающий пилот — `lib/comms-agent/adapt-followup-message.ts`, тумблер «Агент коммуникаций» в «Коммуникациях»).

Обоснование (три причины, все подтверждены сегодняшним днём):
- **Надёжность**: сегодняшний сбой Anthropic API на несколько часов уже показал — если AI недоступен, критический путь не должен останавливаться целиком. Держать решения о переходах на детерминированных правилах — это защита от каскадного отказа при следующем сбое AI.
- **Правовая обязанность**: отказы кандидату всегда буквальны по шаблону — это осознанное юридическое решение Юрия, агент не должен иметь возможность его нарушить.
- **Объяснимость**: детерминированный гейт («балл ≥ порога») можно предъявить HR и, при необходимости, кандидату; решение AI — нет.

### 3. UX-фикс панели редактирования стадии «Воронки v2»

Независимо подтверждено координатором и Fable 5 (второе мнение): секция «Тип этой стадии» сейчас — горизонтальный ряд из 11 pill-кнопок внутри боковой панели, визуально неотличимый от навигационных табов, что создаёт путаницу с соседним левым списком стадий (тоже похож на список-навигацию, но означает совсем другое — реальную последовательность шагов ЭТОЙ воронки).

Принцип: частое действие — крупный интерактив, редкое — компактное поле-значение. Тип стадии задаётся практически всегда один раз, при создании.

- Полный выбор из 11 типов (сетка карточек) — только на экране **«Добавить стадию»**.
- Внутри панели уже существующей стадии — тип превращается в компактный `<Select>` в шапке панели («Тип: Демонстрация ▾»), как остальные выпадающие поля той же панели (например «Гейт по баллу»).
- На экране остаётся ровно одна списочная навигация — левый список стадий воронки.

## Не входит в этот дизайн (сознательно отложено)

- Полная миграция существующих вакансий со старого конструктора/легаси-пайплайна на «Воронка v2» — объём выясняется на этапе планирования (сколько вакансий затронуто, какие данные переносить).
- Расширение возможностей агента за пределы адаптации текста дожимов (Фаза 2/3 каталога сообщений) — отдельная, уже идущая параллельно линия работы (не блокируется этим дизайном).
- Слой мониторинга (детектор сбоя AI, сверка стадии с hh, проверка «слепого инвайта без балла») — реализован отдельно сегодня же (`feat/watchdog-ai-and-hh-reconcile`), не часть этого дизайна, но является предпосылкой доверия к решениям на правилах (гейты должны быть видимы, если они перестали срабатывать).
- AI-фолбэк для резолвера имени кандидата в сомнительных случаях (обсуждалось отдельно, зафиксировано в памяти координатора) — не относится к воронке напрямую.

## Критерии успеха

- Есть ровно одно место, где HR настраивает путь кандидата по стадиям — вкладка «Воронка v2».
- Первое приглашение всегда попадает в «Первичный контакт» на hh, без возможности тихого дрейфа настройки.
- Агент коммуникаций документированно и технически не может: (а) менять стадию кандидата, (б) отправлять текст отказа не по шаблону.
- Панель редактирования стадии не создаёт визуальной путаницы с левым списком стадий.
