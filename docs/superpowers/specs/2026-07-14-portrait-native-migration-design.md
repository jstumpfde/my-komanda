# Дизайн: нативный перенос функционала Портрета в Воронку v2

Дата: 2026-07-14. Статус: утверждён Юрием (координатор-сессия, 6:00 МСК), готов к планированию.

## Проблема

Портрет вакансии — тщательно настроенный набор AI-скоринга, авто-приглашения,
дожима, автоответов и уведомлений — существует ОТДЕЛЬНО от Воронки v2. На
13.07 «Стадия 1» Воронки v2 сделана read-only сводкой Портрета (карточка со
ссылкой «Открыть Портрет», см. `docs/superpowers/specs/2026-07-13-funnel-v2-consolidation-design.md`)
— это временное решение, ЧТЕНИЕ данных, а не их перенос.

Юрий явно просил (13-14.07): всё, что сегодня настраивается в Портрете НИЖЕ
блока «Автоматический отбор по баллу», должно стать **собственными,
редактируемыми полями внутри самой Воронки v2** — не ссылкой на Портрет, а
нативной частью конструктора. Портрет при этом пока НЕ трогаем (ничего не
удаляем, не мигрируем данные) — это чисто аддитивная работа: строим
параллельную нативную конфигурацию в Воронке v2.

## Что переносится и куда — итоговая раскладка

Согласовано в переписке 14.07 (координатор + Юрий), категория по каждому
пункту функционала Портрета:

### → Стадия 1 (Отклик → приглашение на демо)
Из «Автоматический отбор по баллу»:
- Пороги отбора (авто-отказ / авто-приглашение)
- Авто-отказ: текст письма + задержка отправки
- Авто-приглашение: текст приглашения + задержка первого сообщения +
  нерабочее время (вкл/выкл + задержка) + стадия hh.ru при приглашении

(Уже частично сделано 13.07 как read-only чтение из Портрета — см. `feat/funnel-v2-stage1-portrait-parity`.
Эта задача заменяет read-only чтение на собственные поля стадии.)

### → Стадия 2 (Демо 1-я часть → переход на 2-ю часть)
Из «После анкеты → 2-я часть»:
- Пороги теста / AI-оценки ответов (условие ИЛИ)
- Способ перевода на блок 2 (авто / плашка / письмо / комбинация)
- Плашка на блоке 2 (заголовок + текст)
- Текст письма-приглашения на 2-ю часть + задержка перед отправкой
- Экран «Спасибо» для не прошедших (заголовок + текст)
- Действие если не прошёл гейт

### → Коммуникации (общий слой, не привязан к конкретной стадии)
- Стоп-слова → отказ (список + действие при срабатывании + прощальное сообщение)
- Частые вопросы → авто-ответ
- Telegram: подходящие кандидаты (уведомление HR)
- Горячий кандидат стынет (уведомление HR)

### Остаётся в Портрете как источник данных (НЕ дублируется в стадии)
- Структурированные требования к кандидатам (жёсткие/желательно/deal-breakers,
  идеальный профиль, веса скоринга) — это модель расчёта балла, не поведение
  стадии. Стадии Воронки v2 продолжают ссылаться на балл, который считает
  Портрет.
- Общие переключатели дожима (использовать стоп-слова, не дожимать кандидатов
  с Портретом ниже X) — общие настройки, читаемые per-stage дожимом.

### Не переносится вообще (отдельный контур)
- AI-фильтр откликов (legacy, «вне Портрета» по собственной документации кода)

## Архитектура переноса

**Хранение:** новые поля добавляются в `FunnelV2Stage.rule` (тип
`lib/funnel-v2/types.ts`) для per-stage настроек (Стадия 1: задержка/
нерабочее/пороги/тексты; Стадия 2: пороги теста/переход/плашка/письмо).
Коммуникационные настройки (стоп-слова, частые вопросы, TG-уведомления,
горячий кандидат) — общие для вакансии, не per-stage — хранятся на уровне
`vacancies.description_json.funnelV2` (не в конкретной стадии) или в уже
существующей структуре `funnelV2.communications`, если конструктору есть
смысл завести отдельный неймспейс (решается на этапе плана — сверить, нет ли
уже подходящего места).

**Важно — аддитивно, не заменяем:** Портрет продолжает существовать и
работать как раньше для вакансий на легаси-контуре (`funnelV2RuntimeEnabled=false`).
Новые нативные поля Воронки v2 используются ТОЛЬКО когда движок v2 включён
(сейчас — owner-only тумблер). Никакой миграции существующих данных Портрета
"на слом" не делается — это чисто новая, параллельная конфигурация.

**Инициализация значений:** когда HR впервые открывает Стадию 1/2 в
конструкторе Воронки v2 БЕЗ ранее сохранённых нативных значений — поля
предзаполняются текущими значениями Портрета (копия на момент открытия, не
живая ссылка) как удобная отправная точка, дальше редактируются независимо.
Это НЕ синхронизация — после первого сохранения Стадия 1/2 живёт своей
жизнью, изменения в Портрете больше не влияют на неё (в отличие от вчерашнего
read-only-чтения, которое ВСЕГДА показывало живое значение Портрета).

**Рантайм:** `lib/funnel-v2/runtime-executor.ts` и точка входа в
`lib/hh/process-queue.ts` (гейт v2) читают значения из
`FunnelV2Stage.rule`/`funnelV2.communications`, а НЕ из `getSpec()` (Портрет) —
как это временно было сделано 13.07 для задержки/нерабочего времени. Это
меняется на чтение из нативных полей.

## Не входит в этот дизайн (сознательно отложено, по прямому указанию Юрия 14.07)

- Удаление дублирующих секций из Портрета — явно НЕ делаем сейчас
  («ничего никуда не переезжает»). Останется отдельной будущей задачей после
  того, как нативные поля Воронки v2 обкатаны и Юрий убедился, что они
  полностью покрывают функционал.
- Миграция существующих вакансий/данных — не требуется, т.к. Портрет
  продолжает быть источником истины для легаси-контура.
- AI-фильтр откликов (legacy) — не переносится, отдельный контур.

## Критерии успеха

- Стадия 1 и Стадия 2 конструктора Воронки v2 имеют собственные редактируемые
  поля для всего перечисленного выше — без ссылки «читать из Портрета».
- Коммуникационные настройки (стоп-слова, частые вопросы, TG, горячий
  кандидат) редактируются в самой Воронке v2 (или в «Коммуникациях», если так
  удобнее архитектурно — решить на плане), не только в Портрете.
- Портрет не тронут, легаси-контур работает как раньше.
- Рантайм Воронки v2 при включённом движке использует нативные значения, не
  Портрет.
