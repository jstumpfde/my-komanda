# Модульная платформа — план перехода

> Дата: 2026-06-17. Основано на двух многоагентных анализах по реальному коду
> (архитектура модулей + гранулярный entitlement). Решения Юрия:
> деплой = «изоляция риска + парсеры/агенты как отдельные воркеры» (Фазы 0–3);
> сначала этот план-документ, потом реализация.

## 0. Проблема и цель

**Боль:** продуктов/модулей много (~15+), у Маркетинга 10-20 под-фич. Правка одного
модуля = пересборка всего монолита; были инциденты 502 / build-lock. Хочется:
модули разрабатывать/тестировать/чинить/катить **независимо в течение дня**, потом
«подключать» к платформе; клиенту в кабинете включать только нужное — **вплоть до
одной атомарной под-фичи**, и тогда он видит в своей платформе **только её**.

**НЕ делаем:** Module Federation / микро-фронтенды. Анализ оценил как
непропорционально дорого для Next 16 App Router + 1 сервер + малой команды
(App Router их из коробки не поддерживает). Балл 81 у модульного монолита против
проигравших микро-фронтендов.

## 1. Целевая архитектура

Четыре слоя поверх **одного ядра** (модульный монолит):

1. **Каталог** — единый реестр `модуль → фича` с манифестами (что есть, где живёт,
   сколько стоит, нужен ли воркер).
2. **Права (entitlements)** — гранты на уровне **фичи** per-тенант (+ лимиты).
   `modules`/`tenant_modules` остаются слоем лицензии/биллинга; `enabled_modules` —
   кэш видимости; источник правды для навигации — entitlements.
3. **Кабинет** — сайдбар / роуты / домашняя строятся динамически из активных
   entitlements. Один грант = «киоск» (фича как корень приложения, без чужого меню).
4. **Исполнение** — UI-модули в монолите; **тяжёлые фоновые модули
   (парсеры/агенты) — отдельные деплоимые worker-процессы** за общей очередью.

## 2. Что УЖЕ есть (переиспользуем, сокращает работу)

| Что | Где | Роль в плане |
|---|---|---|
| Реестр модулей | `lib/modules/registry.ts` (`MODULE_REGISTRY`, 13 модулей) | расширить до уровня «фича» |
| Лицензия + лимиты | таблицы `modules`, `tenant_modules` (isActive, maxVacancies/Candidates/Employees/Users, `customLimits` jsonb), `lib/modules/access.ts` (`checkModuleLimit`) | слой биллинга/лимитов; добавить грань фичи |
| Видимость | `companies.enabled_modules` jsonb, уже в JWT (`auth.ts`), live без релогина | станет проекцией entitlements |
| Грубый гейт | `middleware.ts` (opt-in `MODULE_GATING_ENABLED`, runtime=nodejs, grandfather) | + карта `feature → path` |
| Точный гейт-прецедент | `requireDirector` (26 мест) | образец для `requireFeature` |
| Адаптер-реестр | `lib/channels/types.ts` (`ChannelAdapter`: send/parseInbound) | образец для парсер-фреймворка + воркеров |
| Cron-инфра | `cron_runs`, `startCronRun/finishCronRun`, cooldown/activity-guard | основа очереди задач воркеров |

## 3. Модель данных (новое)

### 3.1 Каталог фич — `module_features`
```
module_features (
  id            uuid pk,
  module_id     text not null,          -- ссылка на MODULE_REGISTRY id ('marketing')
  feature_key   text not null unique,   -- 'marketing.competitor_analysis'
  name          text not null,          -- 'Конкурентный анализ'
  root_href     text not null,          -- '/marketing/competitors' — корень фичи
  status        text not null default 'active', -- active|beta|hidden
  worker        text,                   -- null = UI-only; иначе id воркера
  billing       jsonb,                  -- { sku, priceRub, metering: 'flat'|'usage', creditCost }
  sort_order    integer default 0
)
```
Источник правды каталога — **манифест в коде** (см. §4); таблица — проекция/кэш
для админки и джойнов.

### 3.2 Права на фичу — `tenant_features`
```
tenant_features (
  id           uuid pk,
  tenant_id    uuid not null,
  feature_key  text not null,           -- = module_features.feature_key
  is_active    boolean default true,
  limits       jsonb,                   -- per-feature лимиты/квоты (перекрывают tenant_modules)
  source       text not null,           -- 'sku' | 'manual' | 'grandfather'
  expires_at   timestamp,               -- null = бессрочно
  unique (tenant_id, feature_key)
)
```

### 3.3 Производные правила
- **Модуль активен у тенанта ⟺ есть ≥1 активная `tenant_features` его фич** →
  убирает «модуль в меню, а внутри всё пусто/locked».
- `enabled_modules` пересчитывается из entitlements (обратная совместимость).
- **Grandfather:** существующим тенантам при миграции выдаём фичи всех их активных
  `tenant_modules` (никто ничего не теряет).

## 4. Манифест модуля (формат)

Расширяем `ModuleConfig` (сейчас плоский `menuItems`) полем `features`:
```ts
interface FeatureManifest {
  key:        string            // 'marketing.competitor_analysis'
  name:       string
  rootHref:   string            // корень фичи (для resolveHomeRoute)
  paths:      string[]          // префиксы страниц фичи (для навигации/гейта)
  apiPrefixes: string[]         // префиксы API (для requireFeature)
  billing?:   { sku: string; priceRub: number; metering: 'flat'|'usage'; creditCost?: number }
  worker?:    string            // id воркер-сервиса, если фича фоновая (парсер/агент)
}
```
Меню строится из `features` (не из ручного `menuItems`); legacy-`menuItems`
постепенно сворачиваются в фичи (переходный период — см. риски §10).

## 5. Навигация и «киоск» (динамический кабинет)

- `resolveNavShape(entitlements)` → `FULL` | `SINGLE_MODULE` | `SINGLE_FEATURE`.
- `SINGLE_FEATURE` (киоск): сайдбар **не монтируется**, фича открывается как корень.
- `resolveHomeRoute(entitlements)` **на сервере** заменяет жёсткий
  `router.replace('/hr/vacancies')` в `app/page.tsx:59` (корень бага «купил Базу
  знаний → попал в пустой HR»). Нужен `primary_feature_key` для мгновенного
  first-redirect без мелькания чужого HR.
- locked / up-sell пункты показываем **только в `FULL`**; в киоске их нет.
- **Сайдбар свести в один layout** — сейчас монтируется в ~30 страницах; без единой
  точки монтирования киоск protекает (см. риски).

## 6. Серверная защита — `requireFeature`

Два уровня: грубый в `middleware` (из JWT-entitlements, быстро) + точный
`requireFeature(featureKey)` в API-роутах (из БД, точно; `/api` middleware не
покрывает).

**537 `route.ts`, сейчас feature-гейта нет нигде.** Стратегия:
1. Скрипт-инвентаризация `path → feature_key` (карта покрытия, чек-лист).
2. Grandfather: старые тенанты = все фичи активных модулей → ничего не ломается.
3. Проставлять `requireFeature` по модулям, начиная с **продаваемых отдельно**.
4. Тест-карта покрытия роутов (CI) — забытый роут = тихая дыра (Turbopack не
   тайпчекает), поэтому покрытие проверяем автоматически.

## 7. Worker-сервисы (парсеры/агенты) — отдельный деплой ⭐

> Это ключевой выбор Юрия: тяжёлое фоновое чиним/катим днём, не трогая монолит.

**Кандидаты в воркеры:** парсеры (лиды / сайты / маркетплейсы / конкуренты),
агент рекламы (Яндекс.Директ), дозвонщик (`dialer`), ОКК (`qc`) — всё, что по
природе фоновая задача, а не UI.

**Контракт платформа ↔ воркер** (по образцу `ChannelAdapter` + `cron_runs`):
- Таблица очереди `jobs (id, tenant_id, feature_key, worker, payload jsonb,
  status, result jsonb, attempts, scheduled_at, ...)`.
- Платформа кладёт job → воркер (отдельный процесс) поллит/слушает → пишет
  `result` + опц. HTTP-callback. Монолит job-ы только создаёт и читает результат.
- Воркер живёт **отдельным PM2-процессом** (позже — отдельным репо), читает ту же
  БД/очередь. **Правка/деплой воркера НЕ пересобирает монолит.** Падение воркера
  не роняет UI; версия воркера — в манифесте.

**Единый парсер-фреймворк + адаптеры** (`lib/sources`), подтверждённый прецедентом
`lib/channels`:
```ts
interface ParserAdapter {
  source: string                         // 'hh' | 'avito' | 'ozon' | 'wb' | 'site' | ...
  fetch(creds, query): Promise<RawItem[]>
  normalize(raw: RawItem): LeadOrProduct
}
```
Один движок (расписание, прокси/антибот, хранилище, нормализация) + per-источник
адаптеры/конфиги → НЕ 20 кодовых баз. Дубли `hh`/`avito` scan-incoming свести сюда
(через shadow — §10).

## 8. Биллинг микро-пакетов

- **SKU** = набор feature-grants + лимиты; **минимальный SKU = одна фича** с
  маленьким чеком. Разворачивается в записи `tenant_features`.
- Метринг «по количеству» — через **кредиты** (см. концепт токенов/кредитов):
  списание за операции воркера/AI; `billing.metering = 'usage'`, `creditCost`.
- Цена/тип — в `billing` манифеста фичи (§4), редактируется в админке.

## 9. Поэтапный план

| Фаза | Что | Оценка | Риск | Катится отдельно |
|---|---|---|---|---|
| **0** | Жёсткий `tsc --noEmit` гейт в CI (`exit 1`) + `error.tsx` на каждый модуль | 0.5–1 д | ~0, без смены поведения | ✅ да |
| **1** | Манифест `features` + каталог `module_features` + сидинг из текущих `menuItems` | 3–5 д | low | ✅ да |
| **2** | `tenant_features` + entitlements в JWT + динамический кабинет + `resolveHomeRoute` + свод сайдбара + киоск + `requireFeature` (поэтапно, grandfather, флаг + shadow) | 1.5–2.5 нед | medium | за флагом |
| **3** | Worker-фреймворк `lib/sources` + очередь `jobs` + первый воркер-парсер (отдельный PM2); консолидация hh/avito через shadow | 1–2 нед на первый | medium | ✅ воркер — да |
| **4** | Биллинг: SKU + кредиты/метринг | отдельно | medium | — |

## 10. Риски (из дилбрейкеров судей — держать в требованиях)

- **tsc-гейт обязан быть жёстким** (`exit 1`), не «мягким/мониторим» — иначе плацебо,
  вся выгода по корню крашей обнуляется.
- **Киоск требует единой точки монтирования сайдбара** (сейчас ~30 страниц) — иначе
  забытая страница покажет соло-клиенту чужой сайдбар (мелькание/утечка навигации).
- `resolveHomeRoute` **только server-side** — иначе соло-клиент мелькает в чужом HR.
- **`requireFeature` на 537 роутов:** забытый = тихая дыра в авторизации (Turbopack не
  тайпчекает) → нужна автоматическая тест-карта покрытия, не «вручную».
- **Переходный период:** модуль-как-одна-фича-обёртка не показывать как отдельный
  навигационный пункт, пока не развёрнут в реальные атомарные фичи.
- **Консолидация парсеров повышает blast-radius cron-слоя** до докатки → строго через
  shadow + не удалять старый путь, пока новый не подтверждён.
- Монолит остаётся **single-failure-domain** для краша в Server Component / shared-слое
  (db/auth/middleware) — `error.tsx` ловит только render/client. Полную изоляцию даёт
  только вынос в воркер (§7), что и делаем для тяжёлого.

## 11. Вопросы — статус (обновлено 17.06)

1. **Грань прав** — ⬜ открыт. Рекомендую **плоские ключи `module.feature`** с
   модулем-родителем в манифесте (проще гейт и кабинет, иерархия выводится из
   префикса). Подтвердить при Фазе 1.
2. **Deploy-канал** — ✅ РАССЛЕДОВАНО. Боевой канал = ручной `/root/deploy-prod-safe.sh`
   (сборка НА сервере, `next start` fork-mode, `pm2 reload`). CI-канал `deploy-prod.yml`
   (off-box build, workflow_dispatch) **построен, но ни разу не запускался** (gh run list
   пуст), недонастроен (нет секретов NEXTAUTH_SECRET/DATABASE_URL), `ecosystem.config.cjs`
   (standalone+cluster) спящий (процесс в fork). → **Фаза 0:** довести off-box CI до
   боевого (2 секрета + жёсткий tsc-гейт + обкатка dispatch на стейджинге), оставить
   `deploy-prod-safe.sh` как fallback. Это и есть «безопасный + независимый» деплой.
3. **Первый воркер-пилот** — ✅ **hh-парсер** (уже готов в `lib/hh`). Фаза 3: обернуть
   готовый парсер в worker-фреймворк (`lib/sources` + очередь `jobs`), не переписывая логику.
4. **Маркетплейс/ecommerce** — отложить. ВМЕСТО: на отдельной платформе **«Market Radar»**
   (другой сервер) уже частично реализован модуль **«Конкуренты»** — вытащить его и
   прикрутить к основной платформе как фичу `marketing.competitors`. Нужен доступ к
   коду Market Radar (git-репо read-only ИЛИ ssh на тот сервер) → смаппить модуль
   (роуты/lib/схема/интеграции/зависимости) → план извлечения+интеграции → перенос.
