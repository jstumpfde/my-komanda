## Резерв (готовность ~60%)
Есть: таблицы (talent_pool_entries + candidates.stage='talent_pool' + talent_campaigns),
жизненный цикл архив→корзина→удаление (cron talent-pool-cleanup), failAction='reserve'
в движке v2 уже кладёт кандидата в резерв, страница /hr/talent-pool с табами.
Дыры: правил авто-добавления нет в UI (S–M: вывести reserve в зоны Воронки 3);
кампании прогрева НЕ отправляют сообщения (L: связать с каналами email/tg/hh);
попадание в резерв не запускает кампанию (M); ответы не трекаются (M).

## Интервью (готовность ~75%)
Есть: таб «Интервью» на вакансии (InterviewsView), calendar_events с полями
interviewOutcome/Rating/Decision/Notes, фиксация исхода в карточке кандидата
(состоялось/неявка/перенос + звёзды 1–5 + решение advance/offer/reject/reserve),
крон напоминаний (24ч/утро/2ч/1ч: in-app + TG + hh-чат кандидату).
Дыры: назначение интервью НЕ переводит кандидата на стадию interview автоматически (M);
решение по итогам НЕ двигает кандидата по воронке (L: decision→offer_sent/rejected/
talent_pool, при v2 — через advance-stage); итоги не в отчётах (M); связь с
candidate_contacts не пишется (S).

## hh-стадии и отказы
Есть: пуш стадий (invite/assessment/interview/consider/discard через
changeNegotiationState + trySync* в sync-stage.ts), входящий синк
sync-inbound-stages (discard_by_applicant → rejected + initiator='candidate'),
отложенные отказы (pending-rejections: задержка rejectionDelayMinutes, рабочие
часы, санитайзер нейтральных текстов) — инварианты «не мгновенно» и «не
настоящая причина» УЖЕ в коде.
Открытия по доке hh (spec в скратчпаде hh-spec.json):
- **offer и hired МОЖНО ставить через API** (наш stage-mapping.ts:20 ошибочно
  считает, что нельзя — дешёвый фикс, закрывает синк Оффер/Нанят);
- есть bulk-перевод до 50 откликов за вызов (для «закрыть вакансию → отказать
  оставшимся»);
- кастомные под-стадии: ЧИТАТЬ можно (funnel_stage.substate, sub_collections),
  запись спекой не подтверждена — нужен 1 живой тест-перевод;
- справочника ПРИЧИН отказа у hh НЕТ (только message) — причина остаётся нашей
  внутренней таксономией (уже есть + отчёт), в hh уходит нейтральный текст;
- причину отказа кандидата hh не отдаёт — только его сообщения в чате.
Дыры: **cron hh-negotiation-sync НЕ включён в crontab** (входящий синк мёртв —
включить = 1 строка, решение Юрия: меняет стадии живых кандидатов);
offer/hired не пушатся; «закрыть вакансию → отказать оставшимся» отсутствует
(дёшево: массовый pendingRejection с разбросом — все гварды переиспользуются);
причина отказа не обязательна в bulk/зонах/стоп-факторах.
