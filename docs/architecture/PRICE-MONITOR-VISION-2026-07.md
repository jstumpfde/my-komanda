# Мониторинг цен → инструмент рыночной аналитики Airbnb (видение Юрия 05.07.2026)

Уточнение Юрия: это не «таблица цен», а инструмент, который показывает **что
реально видит гость**, следит за рынком вперёд на 3–6 месяцев и объясняет, чем
конкурент привлекательнее. Рынок — Пхукет (не РФ), эталон — Mida Grande Resort.

## Ключевая боль (главная ценность продукта)
Хозяин задаёт цену в приложении Airbnb, но **не видит, какую цену показывают
гостю** в поиске (Airbnb накручивает сборы, smart pricing, скидки за срок).
Мы показываем именно **гостевую цену** (из search/PDP) — это и есть решение.
Раздел: «я задал свои объекты → вижу, что показывается, когда меняю цену».

## Что подтверждено извлекаемым (смоук 05.07, sidecar на Риге)
- **Гостевая цена** за диапазон дат со скидками — search + /price. ✅ (уже в Ф1)
- **Доступность на 365 дней вперёд** — /calendar, 1 запрос. ✅
- **Цены вперёд (3–6 мес)** — per-day цену Airbnb анонимно НЕ отдаёт; берём
  точечно /price по датам (семплинг раз в неделю × периоды). ✅ (дороже, но ок)
- **Привлекательность листинга** — /details: фото (у теста 66), рейтинг по 7
  осям (чистота/локация/цена-качество/точность/коммуникация/заезд/итог),
  отзывов, суперхост, гость-фаворит, home_tier, тип, вместимость, удобства
  (парсинг кол-ва удобств — доработать). ✅
- **Позиция в выдаче** — из порядка результатов search, пишем во времени. ✅
- **Привязка аккаунта** — ✅ РЕШЕНО (05.07). Ссылка на ЛИЧНЫЙ профиль
  (users/profile/NNN) анонимно закрыта (PERMISSION_DENIED), НО: по любому
  room-URL хозяина → get_details → host id из reviewee.id отзывов (host.id
  пустой, но reviewee = хозяин) → get_listings_from_user(hostId) отдаёт ВСЕ
  объявления. Юрий (host 128074018) = 11 юнитов Mida Grande с рейтингами/фото.
  Sidecar POST /host-listings {room_id} → {hostId, count, listings[]}.

## Уточнения к модели «наш объект ↔ конкуренты»
- **ЖК определяем НЕ по имени** (имя у юнитов комплекса разное). Комплекс =
  курируемый НАБОР листингов: радиус ~50–150 м вокруг объекта + ручной выбор
  «это мой ЖК» из выдачи. Имя-фильтр (complexFilter) — опциональная доп.
  эвристика, не основной механизм.
- **Два режима поиска, оба нужны:** (а) по ЖК (набор листингов комплекса),
  (б) по местности (радиус ~1 км, весь район) — сравнение «по уровню» рынка.
- **Мониторинг новизны:** какие объявления появились новые в районе/ЖК и по
  какой цене (first_seen_at уже есть; вывести «Новые за неделю» + сигнал).

## Конструктор сравнений (табы)
Владелец сам собирает группы сравнения, каждая = свой таб:
- Таб «Район» — радиус, все листинги местности.
- Таб «Мой ЖК» — курируемый набор комплекса.
- Таб «Другой комплекс N» — добавляемые чужие комплексы, чтобы понимать рынок
  в целом и ценовой спред между комплексами.
Табы добавляются/удаляются; каждый постоянно мониторится, новые подсвечиваются.

## Фазы
- **Ф1 — НА ПРОДЕ (05.07):** объекты по room-URL, конкуренты радиус+имя,
  таблица цен по периодам 7/14/28/30, дельта к медиане, срезы, крон */15, THB.
- **Ф2 «Гость видит + горизонт вперёд»:**
  - Явный фрейминг «Цена, которую видит гость» (копирайт + подсказки).
  - Вид «Цены вперёд 3–6 мес»: доступность-календарь (365 дней) + семплинг
    гостевых цен раз в неделю по горизонту; график/таблица по неделям.
  - Комплекс по курации: complexFilter опционален; ручной набор «мой ЖК» из
    результатов радиуса (уже есть ручное add/ignore конкурентов — достроить UX
    «пометить как мой комплекс»).
- **Ф3 «Конструктор сравнений»:** таблица comparison_groups (тип: район/ЖК/
  другой-комплекс, определение), табы в UI, спред и позиция «по уровню»
  (перцентиль) внутри группы, «Новые объявления».
- **Ф4 «Аналитика привлекательности»:** захват фото/рейтинг/бейджи/удобства/
  позиция в выдаче в снапшоты; таблица «почему конкурента показывают чаще»;
  тренд позиции во времени; сигнал «конкурент обновил цену/фото».
