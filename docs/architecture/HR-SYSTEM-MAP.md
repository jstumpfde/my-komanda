## 1. Сквозная цепочка кандидата (как есть сейчас)

```
ВАКАНСИЯ (конструктор/анкета: 8 секций, descriptionJson.anketa)
   │   ├─ публикация на hh.ru  ◄──► импорт с hh.ru (AI-разбор описания → поля)
   ▼
hh-ОТКЛИК → process-queue → СОЗДАЁТ КАНДИДАТА (stage=new), извлекает резюме
   ▼
① СТОП-ФАКТОРЫ (hard-отсев ДО скоринга: город/возраст/опыт/формат/гражданство/ЗП/права/job-hopping)
   │        └─ совпало → отложенный отказ (cron в рабочее время)
   ▼ (не совпало)
② AI-РЕЗЮМЕ-СКОРИНГ (resumeScore 0-100, Haiku, ОДИН раз) → пороги (lower/upper)
   │   ├─ < lower  → авто-отказ (если включён) / ручной разбор; спорных бот уточняет
   │   ├─ [lower,upper) → демо / предквалификация / ручной разбор
   │   └─ ≥ upper  → авто-приглашение (если включено) на следующий этап (демо/интервью/видео/звонок)
   ▼
③ ПЕРВОЕ СООБЩЕНИЕ + ссылка на демо (hh-чат) ──► AI-ЧАТ-БОТ ведёт переписку
   ▼                                              (pre-filter → Executor → post-filter → watcher)
④ ДЕМО-страница (кандидат проходит) → stage=demo_opened
   ▼
⑤ ФИНАЛЬНАЯ АНКЕТА (кандидат заполняет) → stage=anketa_filled → surveyResponses
   ▼
⑥ AI-АНКЕТА-СКОРИНГ (aiScore, после демо — АНАЛИТИКА, воронку НЕ двигает)
   │   + post-demo экран: зелёный/жёлтый/красный (по anketaThresholds)
   ▼
⑦ АВТООТВЕТ С ТЕСТОМ (если включён) → тестовое задание
   ▼
⑧ ИНТЕРВЬЮ (календарь, планирование, напоминания) → stage=interview
   ▼
⑨ НАЙМ / ОТКАЗ (причина + инициатор) ──► ОТЧЁТ ПО НАЙМУ (агрегация воронки/отказов/созвонов)
```

**Параллельно поверх цепочки:**
- **Воронка (Funnel Builder)** — какие из 23 блоков включены (вкл/выкл этапов ①–⑨).
- **Портрет (Spec)** — при `portrait_scoring=true` источник критериев/порогов для ②/⑥.
- **AI-чат-бот** — ведёт hh-чат, может двигать стадию (отказ/контакт), уточнять спорных.

## 2. Карта связей (главные рёбра данных)

| Откуда | Куда | Через что |
|---|---|---|
| Конструктор вакансии | hh-публикация | `anketa.hhDescription`, employment/schedule/salary |
| hh-отклик | Кандидат | `hh_responses.localCandidateId` → `candidates`, извлечение резюме |
| Вакансия (критерии) | AI-скоринг | `vacancy_specs.spec` (Портрет) **или** `descriptionJson.anketa.ai*` (legacy) |
| Стоп-факторы | Отсев | `vacancies.stopFactorsJson` → matchStopFactors (ДО скоринга) |
| AI-скоринг | Стадия | `resumeScore`+пороги → `candidates.stage` + `stageHistory` + hh-action |
| Демо/анкета | Стадия + данные | `surveyResponses`, `anketaThresholds` → экран; `post_demo_settings` |
| Воронка | Поведение этапов | `funnelConfigJson` + dual-write в legacy-поля (`aiChatbotEnabled` и пр.) |
| Кандидат | hh | `stage` → `sync-stage` → discard/invitation в hh |
| Стадии/отказы/созвоны | Отчёт | `build-report.ts` агрегирует |

## 3. Узлы напряжения — что решить (как должно быть)

> Это и есть корень «ходим по кругу / настроил — не работает / приписывается лишнее».

**A. ТРИ разных хранилища критериев отбора — не синхронизированы.** 🔴 главный
- `anketa.stopFactors[]` (чекбоксы в Сек.5 конструктора) — HR настраивает ЗДЕСЬ.
- `vacancies.stopFactorsJson` — а движок отсева читает ОТСЮДА (другое хранилище!).
- `vacancy_specs.spec.stopFactors` (Портрет) — третье.
- **Симптом:** HR настроил в одном месте, а отсев берёт другое → «настроил, не работает».
- **Как должно быть:** ОДИН источник правды. Предлагаю — `vacancy_specs` (Портрет) как единственный, остальные — зеркала только на чтение.

**B. ДВА контура скоринга параллельно** (legacy-анкета ↔ Портрет-spec) + dual-write. 🔴
- Каждое сохранение пишет в оба → сложность, рассинхрон, мой «фикс там — а работает тут».
- **Как должно быть:** довести ВСЕ вакансии на Портрет, legacy выпилить (после переноса). Сейчас часть на legacy, часть на Портрете.

**C. ТРИ системы AI-оценки** (резюме двигает воронку / анкета — аналитика / рубрика — тень). 🟡
- Путает: «почему оценка есть, а кандидат не двигается». **Как должно быть:** явно зафиксировать, что РЕШАЕТ только AI-резюме; анкета/рубрика — справочно (или объединить).

**D. Воронка-конструктор — пока витрина.** 🟡
- `funnelRuntimeEnabled` ВЫКЛЮЧЕН у всех → рантайм читает legacy-поля, а не блоки конструктора.
- **Как должно быть:** либо включить Runtime (блоки реально управляют), либо честно пометить «визуальная настройка → пишет legacy».

**E. Полнота импорта с hh** (то, что чиним). 🟡
- Уже: about/обязанности/требования, категория, частота выплат, язык, ОВЗ-сужен, график. Осталось: проверить на реальных вакансиях, что ВСЁ садится и ничего лишнего.

**F. Хвосты-баги:**
- `stageHistory` НЕ пишется при ручном переводе стадии HR (история неполна). 🔴 быстрый фикс.
- **B3** дедуп кандидатов частичный (рецидивы). **B9** две системы статусов (stage + legacy-флаги) — унификация.
- Портрет `anketaThresholds` dual-write односторонний (правка в Spec не возвращается в post-demo).
- post-demo «auto»-режим выключен (только manual).

## 4. Что согласовать (развилки — твоё решение)
1. **Критерии отбора (A):** делаем Портрет единственным источником? (свести стоп-факторы/критерии в одно, остальное — зеркала).
2. **Контуры (B):** курс на «всё на Портрет, legacy убрать»? Или пока держим оба?
3. **Воронка (D):** включаем Runtime (блоки реально рулят) или это настройка-витрина?

После твоих ответов по §4 — фиксирую целевую модель сюда и строим строго по ней, а надзор сверяет.
