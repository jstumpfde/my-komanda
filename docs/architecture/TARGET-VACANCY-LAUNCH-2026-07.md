# Целевая картина: «Вакансия за раз-два-три» (Юрий, 02.07.2026, голосом)

Цель фазы: пройти всю платформу, почистить мусор, упростить и соединить связки
так, чтобы запуск вакансии был конвейером: **бриф → сборка → запуск → агенты
ведут кандидатов сами**. Ниже — структурированная запись цели, маппинг на уже
существующее и очередь работ.

## 1. Конвейер запуска вакансии (целевой путь HR)

1. **Бриф клиента** — голосом или в чате: ассистент (Нэнси) сам задаёт нужные
   вопросы и собирает вакансию из ответов.
2. **Либо импорт с hh**: кинул ссылку на вакансию hh → платформа переимпортирует,
   указывает на ошибки/слабые места, даёт рекомендации «у нас настроено лучше так»,
   применяет.
3. **Шаблоны-связки**: роль-шаблон (критерии Портрета + воронка + тексты + демо +
   тест) применяется одним действием; профиль продукта компании подтягивается.
4. **Запуск**: публикация/привязка hh, воронка включена, агенты-контролёры следят.

**Уже есть (не строить заново):** Нэнси-ассистент (голос/чат), vacancy-advisor,
hh-импорт и интеграция, role_templates (0225) + профили продукта, «Сгенерировать
демо», шаблоны воронок (built-in/company/platform).
**Дыры:** Нэнси не умеет «собрать вакансию брифом под ключ»; импорт-по-ссылке с
рекомендациями не оформлен как один флоу; применение роль-шаблона к вакансии
(ТЗ№3) не сделано; связка «всё одним нажатием» не собрана.

## 2. Стадии: внутри — много, снаружи — понятные группы

Боль: «какие стадии есть — не знаю, как кандидаты двигаются — не знаю».

Целевая модель:
- **Внутри системы** стадий может быть сколько угодно (Воронка 3, реестр типов).
- **Наружу** (канбан, дата-таблица, карточка, отчёты) стадии показываются
  **сгруппированными в понятные макро-этапы**; группировка — по типу стадии,
  настраиваемое имя группы. Кандидат всегда «на этапе N: <название>», видно
  «завершил этап 1 → начал этап 2».
- Вид «дата-таблица»: колонка статуса обязана показывать эти группы по
  фактически построенной воронке вакансии (не глобальный хардкод-список).

Каноническая воронка Юрия (эталон для дефолт-шаблона):
отклик → Портрет-скан резюме (рейтинг) → приглашение блок 1 → блок 2 (склеен или
отдельно) → тест → вопросы → телефонный звонок → видео-интервью → встреча в
офисе → оффер → принял → ждём выхода → нанят. На каждом этапе: отказы (3 зоны) и
дожимы (малые/средние/большие).

**Уже есть:** Воронка 3 (ветка feat/funnel-v3-screen: тумблеры стадий, 3 зоны,
тексты, дожимы на стадию); типы стадий = основа группировки; статус-бейджи в
списке. **Дыры:** этапов «телефонный звонок», «встреча в офисе», «принял/ждём
выхода» нет в реестре типов (добавить: call, office_meeting, accepted/awaiting_start
или параметры interview/offer); группировка в канбане/таблице по воронке вакансии;
B9 (двойная система статусов) — стадию пишет только движок воронки.

## 3. AI в чате кандидата — отвечать по СИТУАЦИИ воронки

Боль-пример: кандидату выслали ссылку №2, он пишет «я уже проходил» — система
молчит, а должна сразу ответить по контексту («это вторая часть, вот почему»).

Цель: чат-агент видит **позицию кандидата в воронке** (какая стадия, какие ссылки
высланы, что пройдено) и отвечает мгновенно и по делу; непонятное — эскалация HR.
**Уже есть:** AI-чат-бот (4-уровневая архитектура), автоответы/FAQ, инбокс (#62).
**Дыры:** боту не передаётся funnel-контекст стадии/ссылок; FAQ перехватывает
предквалификацию (K5); чат-бот ⟷ воронка v2 рассинхронизированы (K6).

## 4. Дожимы: система или чат-бот?

Решение (предварительное, Юрий склоняется): **дожимы остаются системными**
(предсказуемо: видно какие касания и когда уйдут — «настроил и видишь»), чат-бот
**реагирует** на входящие поверх них. Бот может получить право «притормозить
дожим» (кандидат ответил — пауза цепочки), но не генерирует касания сам.
Блокер #61 (взаимоисключение) — обязательная предпосылка.

## 6. Резерв (talent pool) — настраиваемый и двухрежимный

- **Правила попадания** настраиваются per-vacancy: диапазон балла (напр. 50–70),
  критерии по любым осям (резюме/анкета/вопросы) → авто-добавление в резерв
  «на следующий раз». В движке v2 уже есть failAction='reserve' — вывести в
  настройки зон Воронки 3.
- **Пассивная стадия** — ожидание (копим, не трогаем).
- **Активные стадии** — аутрич: рекомендованных/сильных хантим сами (пишем,
  интересуемся готовностью) через мессенджеры И hh.

## 7. Исходящий поиск (hh-скан резюме)

Бот ходит по hh по ключевым словам, сканирует резюме ЦЕЛИКОМ (не как hh — по
ключевикам), скорит нашим Портретом и только для достойных тратит платные
открытия контактов (пакеты покупные — беречь лимиты). Посмотреть текущий модуль,
оптимизировать расход открытий.

## 8. Модуль «Интервью»

Кандидаты на стадии интервью автоматически попадают в таб «Интервью»: с кем
назначено / прошло / результат; результат записывается на кандидата и виден в
воронке.

## 9. hh-стадии и отказы (ОЧЕНЬ ВАЖНО — инварианты платформы)

- Проверить, какие стадии hh позволяет двигать через API, и двигать их синхронно
  с нашими (в т.ч. под-стадии — см. HANDOFF: не подтверждены, нужен тест).
- Отказ на hh — с причиной из справочника hh (зарплата/опыт/…): выбирается у
  нас, уходит в hh, копится статистика причин.
- Кандидат сам отказался (+его причина) → отдельный статус и статистика.
- Закрытие вакансии → «отказать оставшимся» (вежливо, каскадом с задержками).
- **ИНВАРИАНТ: никогда не писать кандидату настоящую запрещённую причину**
  (возраст/пол/гражданство и т.п.) — только нейтральные вежливые тексты.
- **ИНВАРИАНТ: никогда не отказывать мгновенно** — сначала «спасибо, рассмотрим»,
  отказ уходит с настраиваемой задержкой (час/день — настройка стадии воронки,
  видна в настройках). Движок отложенных отказов уже есть (pending-rejections).

## 10. Онбординг

После «принят» — онбординг-план для компании: кандидат быстро проходит пробный

## Очередь работ (черновик, приоритет сверху)

1. Воронка 3 → CLEAR гварда → прод (owner) → визуальный ок Юрия.
2. #61 взаимоисключение дожимов (лечит двойные касания сразу).
3. Группировка стадий в канбане/таблице + статусы по воронке вакансии.
4. Funnel-контекст в чат-бота (ответ «я уже проходил» и подобные) + фикс K5.
5. Новые типы стадий (звонок, офис, принял/ждём выхода) в реестр + дефолт-шаблон
   = каноническая воронка Юрия.
6. Роль-шаблон одним нажатием (ТЗ№3) + импорт-по-ссылке hh с рекомендациями.
7. Нэнси-бриф: голосовой сбор вакансии.
8. Консолидация (B9, старый дожим-крон off, funnel-builder v1 в реестр) + чистка
   по аудиту.

(Greenhouse/Lever: stages внутри milestone-групп). Ты пришёл к индустриальному
стандарту сам, и наш движок к нему уже близко — добиваем связки, а не строим с нуля.
