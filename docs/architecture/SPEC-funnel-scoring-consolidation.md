## 1. ПРИНЦИП И ЦЕЛЬ
Сейчас пользователь настраивает поля, которые ни на что не влияют (бейджи врут, v2 не
на входящем потоке, авто-отказ выключен, Funnel Builder не источник правды). Цель:
- **Воронка:** Конструктор воронки [Beta] = ЕДИНЫЙ источник правды и UI; рантайм
  читает его; legacy-таб «Воронка» удаляется после миграции.
- **Скоринг:** один набор критериев → понятный движок → честные бейджи; UI показывает,
  что реально работает.
- **AI-бот:** conversational-слой общения (может заполнять анкету в разговоре) —
  как режим в воронке, без дублей текстов.

Жёсткое требование безопасности: на проде живая вакансия с **1505 кандидатами** и
рабочими cron-ами. Любой шаг — за feature-flag, обратимо, с проверкой паритета.

## 2. ТЕКУЩЕЕ СОСТОЯНИЕ (факты аудита, кратко)
- `funnelConfigJson` хранит только `{type, order, enabled}`; все настройки — в legacy
  (aiProcessSettings, stopFactorsJson, stopWordsJson, firstMessagesChain, finalScreens…).
- Рантайм (process-queue, scan-incoming, cron) читает ТОЛЬКО legacy. Builder/
  funnelBuilderEnabled рантайм НЕ читает (только dual-write enabled-флагов в legacy).
- Скоринг: 4 движка. Входящий резюме — `screenResume` (4 поля AI-профиля, веса
  захардкожены). v2 (`Структурированные требования`) — ТОЛЬКО после демо. Авто-отказ
  отключён (`process-queue.ts:614 if(false)`). Критерии дублируются в 3 местах.
- Бейджи «Нужно для AI-скрининга» на Обязанности/Требования/Портрет/Неприемлемо —
  ВРУТ (входящий скоринг их не читает). `unacceptableSkills` — мёртвое для AI.

## 3. ЦЕЛЕВАЯ АРХИТЕКТУРА

### 3.1 Воронка → Funnel Builder как источник правды
- Каждый блок Builder хранит СВОИ настройки внутри `funnelConfigJson` (или связанной
  структуры), а не в разрозненных legacy-полях.
- Рантайм читает конфиг блоков из Builder.
- Недостающее в Builder (закрыть в Phase 1 — паритет):
  - Блок **«Тест»** (квиз-конструктор со скорингом) — НОВЫЙ (п.61), отдельно от
    «Тестовое задание».
  - Полнота настроек блоков: финальные экраны, стоп-слова, пороги, предквалификация-
    режимы, автоответы, recovery-сообщения, response-timing — всё, что сейчас в legacy.
  - Починить пустую отрисовку Sheet настроек блока (контент не грузится, п.45 — не CSS).

### 3.2 Скоринг → один источник критериев
- Свести 3 представления критериев (AI-профиль кандидата / Структурированные требования
  / стоп-факторы) к ОДНОЙ модели на вакансию.
- Явно определить движок: оставить v1+v2 как этапы (резюме→демо) ИЛИ слить; решить
  судьбу v2 на входящем потоке (сейчас не работает там).
- Стоп-факторы: `stopFactorsJson` остаётся жёстким matcher-источником; `aiStopFactors`
  — мягкий канал (не дубль, но в UI показать как разные вещи).
- Веса/мин.опыт: ✅ ВЕСА УЖЕ ПРИМЕНЯЮТСЯ движком v2 (ai-score-candidate-v2.ts
  computeWeightedScore, контур UI→requirements route→v2 проверен 04.06). Утверждение
  «игнорируются» устарело. v1 (входящий резюме-скрин) холистичен намеренно (§6.1 —
  v2 на входящий не добавляем). Мин.опыт — через стоп-фактор «опыт» (применяется).
- Авто-отказ (`if(false)`): осознанно включить/убрать тумблером (не «висит обещание»).
- Бейджи «Нужно для AI-скрининга»: оставить ТОЛЬКО на полях, что реально читает движок.

### 3.3 AI-бот как conversational-слой (видение Юрия)
- Бот ведёт всё общение в чате, отвечает на сообщения кандидата.
- Если кандидат отказывается от анкеты-формы → бот предлагает задать вопросы и
  **заполняет анкету в разговоре** (мапит ответы в поля анкеты кандидата).
- В воронке: «Сообщения» (шаблоны) vs «AI чат-бот» — взаимоисключающие режимы блока
  общения; тексты-шаблоны не дублировать.

## 6. РЕШЕНИЯ ЮРИЯ (отвечено 03.06)
1. **Скоринг — НЕ трогаем движок в этой консолидации.** Оставляем v1+v2 как есть, v2
   на входящий поток НЕ добавляем (Юрий: «не знаю что это, пока оставим»). НО: бейджи
   «Нужно для AI-скрининга» всё равно привести к правде (показывать только на реально
   используемых полях) — это не про движок, это про честность UI.
2. **Авто-отказ — РУЧНОЙ ТУМБЛЕР (обязательно).** Заменить хардкод `if(false)` на
   настраиваемый тумблер (HR сам включает/выключает; по умолчанию выкл). Не «висящее
   обещание».
3. **Веса — ОСТАВИТЬ** (в UI). → значит сделать так, чтобы веса РЕАЛЬНО применялись
   движком (сейчас игнорируются), раз оставляем. Мин.опыт — оставить аналогично.
4. **AI-бот conversational-анкета — да, финальная фаза** (после набора).
5. **Порядок — на усмотрение Claude.** Решение: сначала консолидация «Найм» (центр.
   узел), платформенный уровень и новый IA — параллельными мелкими треками, где не
   конфликтуют.

### 6.1 🆕 НОВОЕ ТРЕБОВАНИЕ: «Нанять» → переход в воронку Адаптации
В конце воронки найма: отобрали финалистов (прошли тест и т.д.) → кнопка **«Нанять»**
→ кандидат переходит в **СЛЕДУЮЩУЮ воронку «Ввод в должность / Адаптация»** (отдельный
модуль «Адаптация», уже есть в sidebar). То есть найм завершается hire-действием,
которое передаёт человека в онбординг-воронку. Это завершающий блок воронки найма +
мост в модуль Адаптации. (Фаза рядом с консолидацией воронки; учесть в модели стадий.)
