## 1. КАК КАНДИДАТ РЕАЛЬНО ПРОХОДИТ СИСТЕМУ (рантайм, по коду)

Это фактический путь (не то, что обещает UI):

```
hh-отклик → cron hh-import → cron process-queue:
  1. Стоп-факторы (vacancies.stopFactorsJson) — детерминированный matcher
     (lib/hh/process-queue.ts:481 → stop-factors-matcher.ts), ДО AI.
  2. screenResume (lib/ai-screen-resume.ts, Haiku) — скоринг РЕЗЮМЕ.
     Читает РОВНО 4 поля из anketa: aiIdealProfile, aiRequiredHardSkills,
     aiStopFactors, screeningQuestions. Веса захардкожены 40/30/15/15.
     → candidates.resumeScore.
  3. Пороги AI-фильтра (aiProcessSettings.minScoreUpper/Lower = 75/40):
     ≥75 → демо · 40–74 → midRangeAction · <40 → мягкий отказ.
     ⚠️ АВТО-ОТКАЗ ВЫКЛЮЧЕН в коде: process-queue.ts:614 `if (false)`.
  4. Первое сообщение: outbound — scheduleFirstMessagesChain
     (process-queue.ts:907) + дожим (922+).
  → кандидат идёт на демо → отвечает → v1 scoreCandidateById (Sonnet) по
     ответам демо/анкеты → candidates.aiScore. Если есть must-have в
     requirementsJson — параллельно v2 scoreCandidateV2 (A/B, aiScore = v2 ?? v1).
  → финальная анкета → стадии воронки.
```

Чат в hh-чате (scan-incoming.ts:443+): **AI чат-бот идёт ПЕРВЫМ** и короткозамыкает
(если aiChatbotEnabled && есть промпт → processChatbotMessage → continue). Шаблонный
путь («Сообщения») — только fallback, если бот выключен/упал.

## 2. ГЛАВНЫЕ ВЫВОДЫ АУДИТА (с коррекциями моих гипотез)

### A. СКОРИНГ: движков ЧЕТЫРЕ, входящий ≠ то, что показывает UI ⚠️
| Движок | Что | Когда | Источник полей |
|---|---|---|---|
| `screenResume` | резюме | входящий hh-поток | 4 поля AI-профиля (см. §1) |
| v1 `scoreCandidateById` | ответы демо/анкеты | после демо | responsibilities/requirements/anketa |
| v2 `scoreCandidateV2` | резюме vs требования | **только после демо**, при must-have | requirementsJson (Σ=100, deal-breakers) |
| `screenCandidate` | ручной/исходящий | manual/outbound | богатый ScreenInput (НЕ в hh-конвейере) |

**Критично:**
- ⚠️ «Структурированные требования» (таб Дожим: must-have/nice/deal-breaker/веса) =
  вход v2. **v2 НЕ работает на входящем потоке резюме** — только после демо. То есть
  на типичном hh-отклике вся эта секция не влияет на скоринг резюме.
- ⚠️ Бейджи «Нужно для AI-скрининга» на полях **Обязанности / Требования / Портрет
  кандидата / Неприемлемо ВРУТ** для входящего скоринга резюме — `screenResume` их
  не читает (anketa-tab.tsx:1688,1702,1786,1797,1897). `unacceptableSkills`
  («Неприемлемо») — мёртвое поле для AI вообще.
- ⚠️ `aiWeights`/`aiMinExperience` из AI-профиля **игнорируются** `screenResume`
  (веса захардкожены).
- ⚠️ Авто-отказ по низкому скору **отключён** (`if(false)`), хотя UI обещает мягкий отказ.

**Стоп-факторы — НЕ тройной дубль (моя гипотеза п.48 уточнена):**
- `vacancies.stopFactorsJson` — ЕДИНСТВЕННЫЙ рантайм-источник (жёсткий matcher).
- `companies.hiring_defaults_json.stopFactorsDefaults` — только сидинг при создании вакансии.
- `aiStopFactors` (AI-профиль) — отдельный МЯГКИЙ канал (текст в LLM-промпт). Не дубль.

### B. ВОРОНКА: Funnel Builder — это UI-НАДСТРОЙКА, источник правды = legacy ⚠️
⚠️ КОРРЕКЦИЯ моего п.46 (я склонялся «оставить Builder, убрать legacy» — ОШИБКА):
- `funnelConfigJson` хранит ТОЛЬКО `{type, order, enabled}` блоков. Все реальные
  настройки (тексты/пороги/вопросы/стоп-факторы) — в legacy-полях (aiProcessSettings,
  stopFactorsJson, stopWordsJson, firstMessagesChain, …).
- Рантайм (process-queue, scan-incoming, все cron) читает **ТОЛЬКО legacy-поля**,
  `funnelConfigJson`/`funnelBuilderEnabled` не читает НИКОГДА.
- `funnelBuilderEnabled` = UI-флаг + точка dual-write (Builder пишет enabled-флаги
  в legacy, в одну сторону).
- **Нельзя удалять legacy-таб «Воронка» сейчас** — высокий риск: у вакансий с
  `funnelBuilderEnabled=false` (дефолт!) вообще не будет UI воронки; Builder не
  покрывает все настройки (finalScreens, recovery-сообщения и др.).
- Правильный путь: либо довести Builder до полного паритета + бэкфилл всех вакансий
  + переключить рантайм на чтение funnelConfigJson, либо оставить legacy как есть.
  Это БОЛЬШОЙ заход, не «переключить и удалить».
- Баг узкой панели (п.45): ⚠️ классы ширины ИСПРАВНЫ (`w-full sm:max-w-2xl` = 672px,
  funnel-builder.tsx:619). Узость — не CSS, а рантайм (загрузка контента блока через
  отдельный fetch / LoadingSpinner) → нужен браузерный прогон, чтобы найти точную причину.

### C. ОБЩЕНИЕ: «Сообщения работают, AI-бот нет» — уточнение
- scan-incoming приоритезирует AI-бота (короткое замыкание), НО на практике бот не
  включён/без промпта → работает fallback. Рабочий шаблонный путь = **outbound
  first-messages chain + дожим** (process-queue). Это и есть «Сообщения работают».
- ⚠️ «Частые вопросы» (FAQ) — **мёртвы в рантайме**: нигде в lib/ не читаются, только
  ручное копирование HR-ом. (Уточняет п.47.)
- «Хочет созвониться» (callIntent) — подключён, но только режим insist-demo; два
  других режима (slot-and-demo / slot-only) — UI-заглушки.
