# Аудит HR-модуля — логика, дублирование, упрощение

> Дата: 2026-06-10. Автор: Claude (по запросу Юрия).
> Метод: чтение кода (не только UI). Сильные утверждения проверены грепом; спорные помечены «⚠️ проверить».
> Цель: понять, где функциональная настройка переусложнена, и как сделать проще.

---

## 0. Главный вывод

Перегруз — не от числа фич, а от **четырёх системных болезней**, повторяющихся по всему модулю:

1. **Одна сущность настраивается из нескольких мест** с разными моделями данных (воронка, расписание, AI, брендинг, стоп-факторы).
2. **Мёртвые/недопроводённые контролы** выглядят рабочими, но ни на что не влияют.
3. **半-миграции**: старый UI и новый UI живут одновременно, новый «зеркалит» старый (dual-write), источник правды размыт.
4. **Нет единых констант/реестров** (порядок табов, стадии, шаблоны) — всё захардкожено инлайн в нескольких местах.

Ниже — карта, список проблем по приоритету и дорожная карта упрощения.

## 2. Проблемы по приоритету

### 🔴 P1. «Воронка» — три модели в двух местах, часть мёртвая
Подробно разобрано отдельно; ключевое:

| Где | Контрол | Работает? |
|---|---|---|
| Компания `hiring-settings?tab=funnel` | «Дефолтный сценарий» (7 этапов) → `funnelScenario` | ❌ **мёртвый** — поле нигде не читается (проверено грепом) |
| Компания | «Автоматизация» (автодемо/автоинвайт/автоотказ/мин.балл) → `automation.*` | ❌ **мёртвый** — в создании вакансии прямой коммент `// ОТЛОЖЕНО` |
| Компания | «Стадии воронки» (16 платформенных стадий) → `enabledStages/stageOrder/...` | ✅ работает (канбан + действия hh) |
| Вакансия `funnel-builder` | 23 блока + 5 пресетов → `funnelConfigJson` + dual-write в legacy | ✅ работает, **но** `funnelRuntimeEnabled=false` по умолчанию → рантайм читает legacy-поля, конструктор лишь зеркалит |
| Вакансия, скрытые секции | `funnel/messages/followup/aichatbot` — **старый UI тех же настроек**, что и блоки конструктора | ⚠️ дубль: те же `aiScoringEnabled/aiChatbotEnabled/aiProcessSettings` редактируются из 2 UI |

**Две оси смешаны под одним словом «воронка»:** СТАДИИ (где кандидат) и ШАГИ/автоматизация (что делаем). Обе настраиваются и на компании, и на вакансии.

### 🔴 P2. Список кандидатов показывается в 4 местах с разной логикой
1. `/hr/candidates` (все, серверная пагинация, жёсткие колонки)
2. `/hr/vacancies/[id]` таб «Кандидаты» (канбан по стадиям, колонки настраиваемые → **B5** «разные колонки у разных HR»)
3. `/hr/talent-pool` (резерв: merge `candidates.stage='talent_pool'` + отдельная таблица `talent_pool_entries`)
4. Внутри `/hr/candidates?stage=talent_pool` (подмножество #3, но **не видит** ручные записи `talent_pool_entries`)

→ Несогласованные фильтры/колонки (**B5/B6**). Резерв, добавленный CSV, не виден в общем списке. Канбан и таблица — разные компоненты с дублирующей логикой фильтров.

### 🟡 P3. Статусы кандидата — параллельная система ТОЛЬКО в коде (B9), данные чистые ✅понижено
`candidates.stage` в коде поддерживает **16 платформенных слагов** + **legacy-слаги** через `LEGACY_STAGE_LABELS` fallback. **НО проверка прода (10.06): в данных ноль legacy-слагов** — распределение `demo_opened(1260) · primary_contact(1079) · rejected(536) · new(305) · decision(99) · anketa_filled(86) · test_task_sent(64) · test_task_done(12) · interview(10) · ai_screening(3) · hired(1)` — всё это валидные платформенные слаги. То есть данные уже мигрированы; осталась только **мёртвая ветка fallback-кода**. Реальная задача — выпил `LEGACY_*`-кода, а не миграция данных.

Дополнительно — **dead-write**: `lib/funnel/seed-default-stages.ts` вызывается при создании каждой вакансии и компании (`vacancies/route.ts:276`, `companies/route.ts:40,206`). Таблица `funnel_stages` **существует** на проде (вопреки «drop» в 0097 — пересоздана позже), сид в неё пишет, **но читается она только собственным dedup-чеком сида** — больше нигде (`analytics/build-report` матчат строку `funnelStages` по другому поводу). → таблица и сид мёртвые, под снос.

### 🟠 P4. Скоринг кандидата — 4+ параллельных поля
`candidates`: `aiScore, aiScoreV2Details, resumeScore, testScore, rubricScore`. Плюс два конфига порогов: `aiProcessSettings` (v1) и `requirementsJson.scoring_weights` (v2). В проде сейчас 3 параллельных вызова Claude (из бэклога). → одна v2-система + депрекейт старых полей.

### 🟠 P5. Расписание — 3 системы (память велит не объединять, но связь сломана)
1. `vacancies.schedule_*` — окно отправки сообщений (читает `process-queue`). ✅ боевое.
2. `companies.work_schedule_json` (`/hr/settings/schedule`) — рабочее время компании, дефолт для интервью.
3. `/hr/calendar` рабочее время — ⚠️ агент утверждает «хранится только в памяти, теряется при refresh»; **проверить**.
Плюс `/hr/interviews` держит стадии интервью в **localStorage**, рассинхрон с `calendar_events.status`.

Память проекта: «3 системы расписания держать раздельно» — ок. Но: (а) изменения в `/hr/calendar` не влияют на `process-queue`; (б) `/hr/interviews` теряет стадии при refresh. Это не «объединять», а **починить синхронизацию и убрать localStorage-состояние**.

### 🟠 P6. Шаблоны — 5 параллельных систем
| Таблица/хранилище | Что | Уровень |
|---|---|---|
| `demo_templates` | демо/тесты/блоки (length=standard/short/micro) | компания |
| `questionnaire_templates` | анкеты | компания |
| `company_funnel_templates` | конфиг **блоков** воронки (новая таблица, юзается в создании вакансии) | компания |
| `platform_funnel_templates` | блоки воронки, публикует платформа | платформа |
| `companyFunnelPresets` (внутри `hiring_defaults_json`) | пресеты **стадий** (не блоков!) | компания |

→ Два «пресета компании» (`company_funnel_templates` для блоков vs `companyFunnelPresets` для стадий) с похожими именами — главный источник путаницы. Свести в одну систему шаблонов.

### 🟠 P7. Настройки компания↔вакансия — наследование непоследовательное
| Конфиг | Компания | Вакансия | Наследуется? |
|---|---|---|---|
| Расписание интервью | `hiringDefaults.schedule` | `vacancies.schedule_*` | ❌ независимы |
| Стоп-факторы | `stopFactorsDefaults` | `stopFactorsJson` | 🔄 только если `stopFactorsApplyToAll` |
| Стадии/воронка | `hiringDefaults.stage*` | — (нет своей) | глобально от компании |
| AI-пороги | `automation.minScore` (мёртв) | `requirementsJson`/`aiProcessSettings` | 🔄 размыто |
| Брендинг | дефолты компании | `branding_override_enabled` + `description_json.branding` | ✅ явный override |
| Интеграции/источники | компания | `channel_sources`, `hh_vacancy_id` | частично |

→ Нет единого правила «компания = дефолт, вакансия = override». Где-то override, где-то независимо, где-то только компания. Нужен один задокументированный принцип.

### 🟡 P8. Мёртвые/вводящие в заблуждение секции и названия
- `section=ai` называется «AI», а настраивает **расписание** (историческое имя).
- ✅проверено: валидные секции — `page, sources, messages, funnel, funnel-builder, followup, aichatbot, ai, integrations`. `section=schedule`/`section=branding` **не существуют** как секции — любая нераспознанная `section` молча падает на `page` (Брендинг). Это не битые рендеры, а несуществующие URL (моё прежнее предположение было ошибочным).
- Скрытые секции вакансии (`messages/funnel/followup/aichatbot`) дублируют конструктор воронки.
- Заглушки «Скоро» (Авито, SuperJob, Яндекс.Работа, Битрикс24, AmoCRM, dialer) разбросаны по `sources`/`integrations` на двух уровнях.
- `overview`/`overview2` — legacy, под снос.

### 🟡 P9. Дашборд / Аналитика / Отчёт — потенциальное тройное пересечение
`/hr/dashboard` (KPI+воронка сейчас), таб «Аналитика» вакансии (одна вакансия), `/hr/report` (история+share), `/hr/analytics` (вне меню). Цифры должны совпадать. → определить роли: dashboard=сейчас, report=история, analytics вакансии=одна вакансия; `/hr/analytics` верхнеуровневый — вероятно лишний.

## 3. Дорожная карта упрощения (по риску)

### Этап A — «убрать мёртвое» (низкий риск, большой эффект на восприятие)
- A1. Удалить/привязать компанийские секции «Дефолтный сценарий» и «Автоматизация» (обе мёртвые). Если оставлять — привязать к `company_funnel_templates.is_default` и к созданию вакансии.
- A2. Убрать из вакансии скрытые дубль-секции (`funnel/messages/followup/aichatbot`) — оставить только конструктор как единый вход.
- A3. Переименовать `section=ai` → «Расписание»; проверить и починить/удалить битые `section=schedule|branding`.
- A4. Удалить `overview/overview2`, починить или удалить сид в несуществующую `funnel_stages`.
- A5. Спрятать полу-готовые страницы под feature-flag, оставить в меню только боевое.

### Этап B — «сценарий-first» против перегруза 23 блоков (средний риск)
- B1. Сделать выбор сценария основным действием; блоки сгруппировать: **Обязательные(5)** / **Частые(5–6)** / **Расширенные(свернуть)**. Прототип `content_step` убрать из боевого.
- B2. Свести `companyFunnelPresets` + `company_funnel_templates` в **одну** систему шаблонов.

### Этап C — развести две оси и навести порядок в стадиях (выше риск)
- C1. Явно разделить **Стадии** (канбан/статус) и **Шаги** (автоматизация): редактор стадий — один (компания-дефолт + override вакансии), без дубля-блока в конструкторе.
- C2. B9: миграция legacy-слагов `candidates.stage` → 16 платформенных; выпил `LEGACY_*`.
- C3. Единая модель списка кандидатов (один компонент таблицы/канбана + общий реестр колонок/фильтров) для `/hr/candidates`, таба вакансии и резерва (P2, B5, B6).

### Этап D — инфраструктурное (точечно, аккуратно)
- D1. Вынести порядок табов вакансии в одну константу `lib/vacancy-tabs.ts` (B8).
- D2. Завершить или зафиксировать миграцию рантайма воронки (`funnelConfigJson` как единственный источник vs legacy) — убрать половинчатость dual-write.
- D3. Один задокументированный принцип наследования «компания=дефолт, вакансия=override» (P7).
- D4. Унифицировать скоринг на v2, депрекейт старых полей (P4).
- D5. Починить синхронизацию расписаний/интервью (убрать localStorage-состояние в `/hr/interviews`) (P5).

## 4. Проверки (выполнены 10.06, грепом + прод-БД)
- **(P3) legacy-слаги `candidates.stage`:** ✅ на проде ноль legacy — данные чистые, чинить только мёртвый fallback-код.
