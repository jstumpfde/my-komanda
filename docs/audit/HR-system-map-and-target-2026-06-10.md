```

  │    → действия (пригласить/отказать), окно = vacancies.schedule_* + рабочее время
  ├─ pending-rejections (cron, лок 7470002) — отложенные отказы
  ├─ hh-vacancy-sync (cron 05:30) — архив/закрытие вакансий (hh_archived, closed_at)
  └─ outbound sourcing — поиск резюме (платные открытия, AUTO_SCORE топ-N)

AI (ANTHROPIC_API_KEY → CLAUDE_PROXY_URLS Riga / CLAUDE_PROXY_URL CF-worker)
  ├─ скоринг резюме v1 (aiScore) + v2 (aiScoreV2Details) — параллельно, 3 вызова
  ├─ AI-скрининг анкеты (resumeScore/оценка ответов)
  ├─ чат-бот 4 уровня (Pre-filter Haiku → Executor Sonnet → Post-filter → Watcher cron)
  ├─ генерация (описание hh, демо, промпт бота, вопросы скрининга)
  ├─ Нэнси: /api/modules/hr/nancy/chat (502 — хардкод CF-прокси, фикс task_afb099d1)
  │    + публичная Нэнси /ask/[code] (база знаний)
  └─ AI-ассистент вакансии (название/зарплаты/рынок/время публикации, кеш)

Календарь/время
  ├─ companies.work_schedule_json — рабочее время компании
  ├─ vacancies.schedule_* — окно отправки сообщений (читает process-queue!)
  ├─ calendar_events (+переговорные) — события; iCal-экспорт /api/modules/hr/calendar/ical
  ├─ iCal/Google подключение — в настройках /hr/calendar (боевое)

Telegram: эскалации чат-бота, уведомления HR
```

## 2. Принципы целевой архитектуры (из твоих комментариев)

1. **ОДНО место + фильтр/селектор + переиспользуемый компонент** — вместо копий,
   зарытых в каждую сущность (кандидаты, аналитика, стадии, контент, «кого ищем»).
2. **Компания = дефолт, вакансия = осознанный override** — единое правило наследования.
3. **Библиотека = источник, вакансия = reuse** (блоки «О компании», демо, анкеты, тесты).
4. **Воронка гибкая под любые вакансии и ПРОСТАЯ** — сценарий-first, блоки сгруппированы.
5. **Подключение площадок — на компании/платформе, управление — на вакансии** (намеренный сплит).
6. **Разные смыслы времени — раздельно, одинаковые — в одном месте** (разнести по СМЫСЛУ, не по месту).
7. ⭐ **Nancy-first: всё, что можно сделать кнопкой, можно сделать голосом** (см. §4).

## 3. Целевая архитектура: 4 слоя

### Слой 1 — ЯДРО ДАННЫХ: 8 реестров (источников правды)

| # | Реестр | Что содержит | Кто потребляет | Что схлопывает |
|---|--------|--------------|----------------|----------------|
| R1 | **Профиль компании/бренда** | название, описание, логотип, цвета, домены; мультикомпания = список брендов | брендинг вакансии, «О компании», демо, публичные страницы | 3 места брендинга |
| R2 | **Время и люди** | рабочее время компании; календари HR (мульти-HR: личный + общий слот-пул); слоты записи; праздники; переговорные; iCal/Google; назначение вакансия→HR | планировщик интервью, Нэнси-слоты, окно отправки (наследуется вакансией), календарь, interviews | 5 мест расписания, 6 мест интервью |
| R3 | **Стадии воронки** | один реестр стадий компании (16 платформенных, вкл/цвет/hh-действие) + override вакансии | канбан, отчёты, чат-сценарии (смена стадии), интервью-бакеты (проекция) | 2 редактора стадий + localStorage |
| R4 | **Candidate Spec («кого ищем»)** | per-vacancy: hh-поля публикации (единый список навыков как на hh) ⟂ критерии оценки (must/nice/dealbreaker, веса, идеальный профиль, стоп-факторы, мин. опыт) + ЕДИНЫЕ пороги | AI-скоринг (только v2), чат-бот промпт, outbound, предквалификация, стоп-факторы, урок «Кого мы ищем» (публичная проекция) | 4 места критериев, 4 места стоп-факторов, 2 набора порогов, v1/v2 |
| R5 | **Контент-библиотека** | блоки (О компании/О руководителе/…) · демо · анкеты · тесты; аудитория (кандидаты/сотрудники/клиенты); актуальность | таб «Контент», блоки воронки (демо/анкета/тест/видео-визитка), модули за пределами найма | 5 хранилищ шаблонов, 5 «анкет», видео-визитка ×2 |
| R6 | **Чат-сценарии (интенты)** | интент (созвон/зарплата/отказ/вопрос/график/локация/…) → реакция (шаблон + смена стадии + опц. касание); режим: авто (AI чат-бот) / ручной (подсказка-FAQ); защита (injection/abuse) и пороги — общие | scan-incoming, чат-бот (опорный блок — «здесь всё актуально собрано»), FAQ, стоп-слова, «хочет созвониться» | 4 чат-блока, конфликт поведения стоп-слов |
| R7 | **Машина касаний** | ОДИН механизм: цель (открыть демо / дописать анкету / сдать тест / ответить в чате / прогрев резерва) + интенсивность (мягкий/стандарт/актив) + расписание Д+N + ветки шаблонов по состоянию + глушители (стоп-слова, вакансия закрыта, цель достигнута) | дожим демо, дожим теста, напоминания предквалификации, прогрев talent-pool | ≥4 места дожимов, двойная отправка (UI сам предупреждает) |
| R8 | **Реестр сообщений** | типизированные шаблоны с плейсхолдерами: первое сообщение · приглашение на собес · оффер (+эл. подпись) · отказы (нейтральные!) · аварийное · короткие «минутку»; финальные экраны демо | все шаги воронки, чат-сценарии, машина касаний | оффер/приглашение (просьба Юрия), 3 места финал-экранов, тексты отказов в 3+ местах |

### Слой 2 — МАШИНА НАЙМА (рантайм, читает ТОЛЬКО реестры)

**Вертикаль** (путь кандидата) — ~9 шагов вместо 23 блоков:

```
Отклик (hh / форма / outbound / реферал)
  → 1. Фильтр и скоринг  (стоп-факторы из R4 → AI-скоринг v2 из R4 → маршрутизация
        по порогам: демо / предквалификация / отказ / Новый)
  → 2. Первое сообщение  (из R8, окно отправки из R2)
  → 3. Демо              (контент из R5; видео-визитка опц.; финал-экран из R8)
  → 4. Анкета            (шаблон из R5; AI-скрининг ответов по R4 — ЕДИНЫЕ пороги)
  → 5. Тест              (задание из R5 + автоотправка + AI/HR-проверка + касания из R7 — ОДИН блок)
  → 6. Интервью          (слоты из R2: ссылка на календарь ИЛИ Нэнси предлагает слоты)
  → 7. Реф-чек           (вопросы; опц. обязательный)
  → 8. Оффер             (документ из R8 + эл. подпись)
  → 9. Нанят / Резерв    (хорошие не-нанятые → talent-pool, прогрев через R7)
```

**Горизонтали** (вдоль всей воронки): чат-движок (R6) · машина касаний (R7) ·
расписание-гейт (R2) · смена стадий (R3) · уведомления/эскалации (Telegram).

### Слой 3 — ПРОЕКЦИИ (UI = виды на ядро, без собственной логики)

- **Вакансия** = мастер: Анкета (ввод + Spec R4) · Контент (reuse R5) · Воронка
  (сценарий-first: пресет → шаги → горизонтали) · Кандидаты · Аналитика (селектор) · Источники.
- **Кандидаты** = ОДНА богатая таблица (эталон — таб вакансии) + ОДИН drawer (8 табов,
  эталон ширины) везде: /hr/candidates (scope=все), таб вакансии (scope=вакансия),
  talent-pool (scope=резерв). Страницу /hr/candidates/[id] → заменить drawer'ом.
- **Календарь** = центр времени (R2): события, слоты, праздники, iCal, мульти-HR.
  /hr/interviews = фильтр-вид календаря по интервью (бакеты = сохранённые виды, не localStorage).
- **Дашборды по ролям**: /hr/dashboard = live-кокпит HR («кто движется, кто готов») ·
  /hr/report = руководитель + share/TV (будущее: общий TV-борд по отделам) ·
  аналитика вакансии = тот же дашборд с селектором вакансии.
- **Библиотека** = редактор R5. **Talent-pool** = scope-вид кандидатов + кампании (R7) + формы/рефералы.

## 4. ⭐ Nancy-first: голосовое управление как архитектурный принцип

**Требование Юрия:** в будущем весь найм управляется голосом через Нэнси —
«Нэнси, составь вакансию менеджера продаж, зарплата 80–120, воронка с тестом» /
«запусти найм» / «кто сегодня готов к собеседованию?» / «отправь оффер Иванову».

**Что это диктует архитектурно (и почему это ТО ЖЕ САМОЕ упрощение):**

1. **Конфигурация = данные, не экраны.** Нэнси не умеет «кликать по 23 Sheet'ам».
   Она умеет заполнить структурированный конфиг. Значит: один Hiring Config
   (реестры R1–R8) с понятной схемой → его можно заполнить голосом, формой, шаблоном
   или AI-генерацией — РАЗНЫЕ входы в ОДНУ модель.
2. **Слой операций (tools).** Каждое действие HR = именованная серверная операция
   с проверкой прав: `vacancy.create_from_brief` · `vacancy.generate_description` ·
   `funnel.apply_template` · `spec.set_criteria` · `candidate.move_stage` ·
   `candidate.invite / reject / send_offer` · `interview.propose_slots / schedule` ·
   `report.query`. **UI-кнопки и Нэнси вызывают ОДНИ И ТЕ ЖЕ операции.**
   (Сейчас логика частично размазана по клиентским компонентам — её надо опустить на сервер.)
3. **Каждый дубль настроек — двойная цена для Нэнси.** Пока стоп-факторы в 4 местах,
   Нэнси не знает, куда писать «без релокации». Консолидация реестров — не «уборка»,
   а ПРЕДУСЛОВИЕ голосового управления.
4. **Нэнси уже частично есть**: ассистент в /settings/Ассистент, /api/modules/hr/nancy/chat,
   публичная Нэнси /ask, план «Нэнси предлагает кандидатам слоты». Целевое состояние —
   Нэнси-агент с доступом к операциям слоя 4 + контекст из реестров.
5. **Сценарий-уровни Нэнси** (по нарастанию):
   - N1: вопросы-отчёты («сколько откликов, кто ждёт решения») = `report.query` — read-only;
   - N2: действия над кандидатами («пригласи топ-3 на интервью») = candidate.* — с подтверждением;
   - N3: создание вакансии по брифу (голос → Spec → описание → демо → воронка-шаблон → публикация) —
     цепочка операций, draft + ревью HR;
   - N4: автономный найм по заданию («найми 2 менеджеров до конца месяца») — оркестрация
     + контрольные точки. ДАЛЬНЯЯ цель, но реестры/операции делают её достижимой.

## 5. Консолидация 23 блоков воронки → 9 шагов + 4 горизонтали

| Сейчас (23 блока) | Куда уходит |
|---|---|
| AI-скоринг резюме · Стоп-факторы · Предквалификация | Шаг 1 «Фильтр и скоринг» (критерии — из R4, в блоке только пороги/маршруты) |

## 8. Рекомендация: как строить (предложение координатора)

**Центральная конструкция: Вакансия = Бриф → Spec → Сценарий.**
Создание вакансии — три действия, а не 40 настроек:
1. **Бриф** — любой вход: голос Нэнси / текст / импорт с hh / шаблон компании.
   AI превращает бриф в анкету вакансии (hh-поля) + Candidate Spec + черновики
   описания/демо.
2. **Сценарий** — выбор пресета воронки (5 штук); всё работает на дефолтах компании.
3. **Ревью и запуск** — HR смотрит, правит точечно, публикует.
Все остальные ручки существуют, но как override'ы дефолтов, спрятанные за «Расширенные».

**Три уровня конфигурации, одно правило наследования:**
платформа (шаблоны, Битрикс, emergency) → компания (бренд, время, стадии,
сообщения, интенты, дефолт-сценарий) → вакансия (Spec + сценарий + осознанные
override). Настройка ВИДНА с вакансии, но ЖИВЁТ на самом верхнем уровне,
где её достаточно. Один редактор на сущность, рендерится на любом уровне.

**Воронка — «сценарий-first, три горизонта»:**
- Горизонт 1 (90% вакансий): выбрал пресет — поехали (дефолты компании).
- Горизонт 2: 9 шагов вкл/выкл + настройки шага (что специфично вакансии).
- Горизонт 3 «Расширенные»: горизонтали (интенты R6, касания R7, время R2) —
  наследуются от компании, override редкий.

**Стратегия строительства — новое ядро РЯДОМ, не рефакторинг на месте:**
- Реестры R1–R8 + слой операций строятся рядом с legacy; направление dual-write
  разворачивается (сейчас конструктор зеркалит в legacy → станет: реестр источник,
  legacy-поля производные → отмирают).
- Каждый реестр мигрирует ОТДЕЛЬНО (не big-bang), порядок: R4 Spec → R8 сообщения
  → R7 касания → R6 интенты → R2 время → R3 стадии → R5 шаблоны.
- Полигон: вакансия «Помощник по маркетингу» (ИП), Орлинк не трогаем.
- UI переключается фиче-флагом на вакансию-полигон, потом на компанию, потом всем.

**Параллелизация (агенты, непересекающиеся зоны):**
- Агент A: этап A (мёртвое + 10 багов) — можно запускать сразу.
- Агент B: R4 Spec (схема + API + миграция v1→v2).
- Агент C: UI-кит (единая таблица кандидатов + drawer + адаптивная ширина Sheet) —
  чистый UI, ядро не трогает.
- Координатор: ревью, мерджи, деплой, обсуждение концепций с Юрием.

**Решения по развилкам §7 (предложение):**
1. «О компании»: R1 = факты бренда; R5-блок = редакционный текст, генерится из R1.
2. Подробность: hh-описание подробное (для людей/охвата), Spec точечный (для машины);
   AI-ассистент подсвечивает расхождения между ними.
3. Стадии: override = подмножество + переименование с мэппингом на 16 платформенных.
4. Talent-pool: раздел остаётся (база+прогрев+рефералы); формы/трекинг → единый
   механизм «Источники» (один form-builder для вакансии и пула).
5. Контент-шаг-прототип: закрыть; шаг «Демо» берёт ТИП контента из библиотеки R5.
6. Время: календарь = центр; per-vacancy только окно отправки (наследуется).
7. После старта B — этапы C и D параллельно (зоны не пересекаются).

### ⭐ Модель интеграций — ТРИ УРОВНЯ владения (решение Юрия 10.06)
НЕ «свести 4 места в один контур» (отвергнуто) и НЕ «вход/выход» — а по УРОВНЮ владения,
с наследованием вниз. Один и тот же ресурс (напр. Битрикс) виден на разных уровнях по-разному:
1. **ПЛАТФОРМА (админ)** — глобальное подключение ресурса, которое клиент НЕ видит и не настраивает
   (ключи/OAuth-приложение Битрикса, API-ключи площадок на уровне платформы). Живёт в /admin/platform.
2. **КОМПАНИЯ (директор)** — общий дефолт на все вакансии: «наш Битрикс-адрес», общий webhook компании.
   Живёт в /hr/hiring-settings. (У прямого работодателя — обычно один CRM, поэтому дефолт удобен.)
3. **ВАКАНСИЯ** — переопределяет компанию для конкретной вакансии: hh-аккаунт публикации, «кандидатов
   ЭТОЙ вакансии слать в другой CRM/воронку». Живёт в карточке вакансии. Наследует компанию, если не задано.
→ Это тот же модульный принцип, применённый к интеграциям. Текущее «Webhooks+Битрикс в Служебное» —
частично уровень 2, но подключение Битрикса по-хорошему уровень 1, а override на вакансии (ур.3) отсутствует.
Целевое: разложить интеграции по 3 уровням с наследованием, а не переносить наугад. Часть единой
админ-панели платформы + консолидации. (Webhooks/Битрикс убрать из «Служебное» — это не служебное.)

**Двухконтурная схема перехода (решение Юрия 10.06 — не ломать действующее):**
- Контур 1 «Действующий»: заморожен от реформы, только баг-фиксы (этап A) обычным путём.
- Контур 2 «Новое ядро»: ТОЛЬКО новые файлы (`lib/core/…`, `/api/core/…`) и новые
  таблицы; legacy-файлы не редактируются. Пока флаг выключен — код спит, сломать
  действующее не может.
- Переключение флагом `useNewCore` поэтапно: выкл у всех → вакансия-полигон (ИП) →
  ИП целиком → Орлинк → все. Мост: dual-write старое↔новое, откат = выключить флаг.
- Обозначение: код `lib/core` vs legacy; UI-бейдж «Бета» на переключённых; таблица
  соответствия = §5; агентам запрещено править legacy вне баг-фиксов.

**Конечный опыт (к чему всё сходится):**
«Нэнси, найми ассистента маркетолога, бюджет 60–80» → Нэнси создаёт draft
(Spec+описание+демо+воронка), Юрий подтверждает → публикация; найм идёт машиной;
Нэнси докладывает: «3 кандидата прошли анкету выше 80, предлагаю интервью завтра
в 14:00» → «назначай» — слоты из календаря R2.
